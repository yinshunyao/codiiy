# 端到端开发工作流规范

本规则定义从用户对话到交付的完整执行顺序。**任何开发活动都必须按照本流程执行，不允许跳过环节。**

## 场景一：用户提出新需求或功能变更

按以下顺序执行，每一步完成后再进入下一步：

```
① 整理需求(01-or) → ② 开发设计(02-dr) + 测试设计(03-tr) → ③ 代码实现 + 自动化测试 → ④ 归档工作项
```

### ① 整理/更新原始需求
- 检查 `doc/01-or/` 中是否已有该主题的 OR 文档。
- 有 → 将新内容融入对应章节（背景/目标/要求），遵循 `rules-or.md`。
- 无 → 新建 OR 文档，遵循 `rules-or.md` 的命名和章节结构。
- 在 `# 当前工作项` 中登记本次任务。

### ② 同步完成开发设计和测试设计
- 在 `doc/02-dr/` 中创建或更新设计文档，遵循 `rules-dr.md`，必须标注对应 OR 文档路径。
- **同时**在 `doc/03-tr/` 中创建或更新测试设计文档，遵循 `rules-test-design.md`。
- 开发设计和测试设计是**同一步骤的两个产出**，不允许先完成一个再补另一个。

### ③ 代码实现与自动化测试
- 编写功能代码，遵循 `rules-develop.md` 中的编码规范。
- **同步**编写自动化测试代码：
  - 后端测试：遵循 `rules-test.md` 的目录映射规则。
  - WEB E2E 测试：使用 Playwright，代码放在 `test/e2e/`，遵循 `rules-test.md` 中的 E2E 约定。
- 实现前**必须**先确认设计文档已覆盖本次变更。
- 若本次实现影响部署命令、启动参数、环境变量或运行目录约束，必须同步更新 `doc/99-ops/`（至少 `run.md`）。

### ④ 归档工作项
- 将 OR 文档中 `# 当前工作项` 的已完成条目迁移到 `# 已完成工作项`。
- 确认设计文档和测试设计文档与最终实现一致。
- 发布记录采用“草稿先行”策略：细小功能/变更/问题修复先记录到 `doc/99-ops/release-note-draft.md`，不强制实时改 `doc/99-ops/release-note.md`。
- `doc/99-ops/release-note.md` 仅在收到“发布整理”或“发布汇总”指令后，从草稿中提炼并精简更新。

## 场景二：用户反馈 Bug 或问题

按以下顺序执行：

```
① 创建QA文档(91-qa) → ② 分析定位 → ③ 修复代码 → ④ 回填文档 + 补充回归测试
```

### ① 创建 QA 文档
- 在 `doc/91-qa/` 中创建问题文档，遵循 `rules-test-design.md` 中 91-qa 的章节约束。
- 先写 `# 问题描述`，记录现象和复现条件。
- 填写 `# 关联文档`，定位该问题**所属的原始需求（01-or）和开发设计（02-dr）文档**，建立追溯关系。若暂时无法确定，标注"待定位"并在后续步骤中回填。

### ② 分析定位根因
- 排查问题根本原因，在 QA 文档中填写 `# 根因分析`。
- 根据分析结果，**确认并完善** `# 关联文档` 中的文档路径（补充关联的测试设计文档 03-tr 路径）。

### ③ 修复代码
- 若修复涉及设计变更，**先更新** `doc/02-dr/` 中的设计文档，再修改代码。
- 若 Bug 暴露出需求遗漏，需同步在 `doc/01-or/` 中补充需求。

### ④ 回填文档与补充测试
- 在 QA 文档中补全 `# 涉及文件` 和 `# 设计约束更新`（如有）。
- **最终确认** `# 关联文档` 章节完整，包含所有关联的 01-or、02-dr、03-tr 文档路径。
- 在 `doc/03-tr/` 中补充回归测试用例。
- 编写对应的自动化回归测试代码。

## 场景三：仅修改设计文档（无代码变更）

- 修改 `doc/02-dr/` 时，必须同步检查并更新 `doc/03-tr/` 中对应的测试设计。
- 修改 `doc/01-or/` 时，必须检查下游 `doc/02-dr/` 和 `doc/03-tr/` 是否需同步更新。
- 若设计变更包含运维部署影响（启动方式、环境变量、命令参数、目录基准），必须同步检查并更新 `doc/99-ops/`。

## 禁止行为

1. **禁止跳过需求直接写代码**：无论多小的功能变更，都必须先确认 OR 文档中有记录。
2. **禁止跳过设计直接写代码**：代码实现前必须有对应的设计文档覆盖。
3. **禁止设计不同步**：修改开发设计而不检查测试设计。
4. **禁止 Bug 修复不留痕**：用户反馈的问题必须在 91-qa 中有文档记录。
5. **禁止代码无测试**：功能代码提交时必须有对应的自动化测试。
6. **禁止 QA 文档无追溯**：91-qa 中的每个问题文档必须通过 `# 关联文档` 章节关联到对应的原始需求(01-or)和设计文档(02-dr/03-tr)，确保问题可反向追溯到需求和设计。
7. **禁止部署变更不更新运维文档**：涉及启动/部署约束变更时，不得遗漏 `doc/99-ops/` 同步。
8. **禁止发布记录散落**：细小发布项不得分散在聊天记录或临时文件，必须统一沉淀到 `doc/99-ops/release-note-draft.md`。
