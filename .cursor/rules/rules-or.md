# 原始需求 (Original Requirements, OR) 结构与内容标准

当处理位于 `/doc/01-or/` 目录下的原始需求文档时，请遵循以下准则：

## 强制性章节
- 文档**必须**包含五个一级标题，且顺序固定：
  1.  `# 背景`
  2.  `# 目标`
  3.  `# 要求`
  4.  `# 当前工作项`
  5.  `# 已完成工作项`

## 章节内容规范

### # 背景
- 内容应聚焦于**业务上下文**和**用户痛点**。
- 禁止包含技术实现细节或代码设计。

### # 目标
- **必须**使用编号列表（`1.`, `2.`, ...）。
- 每条目标**必须**以动词开头，描述可衡量的用户价值。

### # 要求
- 此部分具有**最高优先级**，任何其他文档或规则不得与之冲突。
- 需明确区分功能性要求与非功能性要求（如错误处理、性能约束）。

### # 当前工作项
- **仅在有进行中的任务时才保留此章节内容**。
- 任务完成后，相关条目**必须**被移至 `# 已完成工作项` 中归档，并清空本章节。

### # 已完成工作项
- 执行当前工作项时参考， 不能 重复执行 本目录下的 工作项
- 主要从 用户对话 或者 `# 当前工作项` 中获取 工作项名称和描述
- 按照 **需求** 和  **问题** 分开组织列举

## 全局原则
- **保持纯粹性**：OR 文档是需求的唯一真实来源，不应混入设计方案。
- **驱动设计**：所有开发工作**必须**依据此文档创建相应《设计文档》。
- **持续同步**：若需求变更，此文档**必须**是第一个被更新的文件。
