# 原始需求 (Original Requirements, OR) 结构与内容标准

当处理位于 `/doc/01 or/` 目录下的原始需求文档时，请遵循以下准则：

## 强制性章节
- 文档**必须**包含五个一级标题，且顺序固定：
  1.  `# 背景`
  2.  `# 目标`
  3.  `# 要求`
  4.  `# 当前工作项`
  5.  `# 已完成工作项`

## 章节内容规范

### # 背景
- 内容应聚焦于**业务上下文**和**用户痛点**。
- 禁止包含技术实现细节或代码设计。

### # 目标
- **必须**使用编号列表（`1.`, `2.`, ...）。
- 每条目标**必须**以动词开头，描述可衡量的用户价值。

### # 要求
- 此部分具有**最高优先级**，任何其他文档或规则不得与之冲突。
- 需明确区分功能性要求与非功能性要求（如错误处理、性能约束）。

### # 当前工作项
- **仅在有进行中的任务时才保留此章节内容**。
- 任务完成后，相关条目**必须**被移至 `# 已完成工作项` 中归档，并清空本章节。

### # 已完成工作项
- 执行当前工作项时参考， 不能 重复执行 本目录下的 工作项
- 主要从 用户对话 或者 `# 当前工作项` 中获取 工作项名称和描述
- 按照 **需求** 和  **问题** 分开组织列举

## 解耦与内聚原则（核心约束）

### 一需求一文档
- **独立的需求必须拆解到独立的文档**，禁止将不相关的需求堆叠到同一个文件中。
- 判定标准：如果两个需求的「背景」和「目标」可以独立描述、互不依赖，则**必须**拆为两个文档。
- 即使属于同一主题（如"标注质检"），不同的变更/需求也应各自独立成文。

### 文档命名规范
- 格式：`【主题分类】具体需求描述.md`（如 `【标注质检】质检整改原因与操作记录.md`）。
- **禁止**使用笼统的名称（如 `标注和质检变更.md`、`XX相关问题.md`），因为这类名称会吸引无关内容不断追加。

### 文档膨胀控制
- 单个 OR 文档的有效需求内容（不含「已完成工作项」）**不应超过 150 行**。
- 当文档超过此阈值时，**必须**审视内容并拆分：
  - 将独立的子需求提取为新文档。
  - 原文档仅保留真正内聚的核心需求。
- 文档内部**禁止**以"追加条目"方式无限增长，每次新增内容时必须重新审视文档结构，确保逻辑清晰、层次分明。

### 禁止变更日志式堆叠
- OR 文档**不是变更日志**。禁止以日期编号（如 `2026031007`、`2026022401`）为一级标题堆叠多个独立变更。
- 每个独立变更应拆分为独立的 OR 文档。
- 如果确实是对同一需求的增量补充，应**融入**原文档对应章节（背景/目标/要求），而非在文档末尾追加新段落。

### 新需求处理决策流程
收到新需求时，按以下步骤判定：
1. **独立性判断**：该需求是否可独立于现有文档描述？→ 是 → **新建文档**
2. **归属性判断**：该需求是否是现有需求的补充说明或约束调整？→ 是 → **融入现有文档对应章节**（非追加到末尾）
3. **膨胀检查**：融入后文档是否超过 150 行？→ 是 → **拆分**

## 全局原则
- **保持纯粹性**：OR 文档是需求的唯一真实来源，不应混入设计方案。
- **驱动设计**：所有开发工作**必须**依据此文档创建相应《设计文档》。
- **持续同步**：若需求变更，此文档**必须**是第一个被更新的文件。
