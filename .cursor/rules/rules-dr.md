# 开发设计文档 (Design Review, DR) 结构与内容标准

当处理位于 `doc/02-dr/` 目录下的开发设计文档时，请遵循以下准则。

## 关联文档（强制）

每个设计文档**必须**包含 `# 关联文档` 章节，用表格列出对应的原始需求和相关文档路径，建立**需求 → 设计**的正向追溯链。

### 格式要求
```markdown
# 关联文档
| 类型 | 文档路径 |
|:---|:---|
| 原始需求 | `doc/01-or/【预标注】预标注采用小模型.md` |
| 关联设计 | `doc/02-dr/01 任务创建/01.02.03 标注配置.md` |
| 测试设计 | `doc/03-tr/06 预标注/预标注业务流程测试设计.md` |
```

### 追溯性约束
- 设计文档中的每一项功能设计**必须**能追溯到至少一条原始需求。
- 若需求变更，先更新 01-or，再同步修改本目录中的设计文档。
- 关联文档表中至少包含一条 `原始需求` 类型的记录；`关联设计` 和 `测试设计` 按实际情况填写。

## 建议章节结构

设计文档根据实际需要组织内容，但**建议**包含以下核心部分（顺序可灵活调整，`# 关联文档` 章节放在文档开头或 `# 功能设计` 之前）：

### 1. 功能设计
- 描述核心业务逻辑、流程、状态流转。
- 涉及前端页面的，说明交互逻辑和页面结构。
- 涉及后端接口的，说明 API 路径、请求/响应格式、关键参数。

### 3. 数据模型（如涉及）
- 表结构变更（新增表、新增字段）使用表格列出字段名、类型、约束、说明。
- 数据迁移策略（若涉及已有数据）。

### 4. 约束与边界
- 非功能性约束：性能、安全、并发、错误处理。
- 明确不做什么（Scope Out），防止设计范围蔓延。

### 4. 关联设计（如涉及）
- 除 `# 关联文档` 表格中的引用外，在正文中说明与其他设计文档的依赖或协作关系。

## 目录组织规范

### 模块化子目录
- 按功能模块建立子目录，子目录使用 `编号 模块名` 格式（如 `01 任务创建`、`06 预标注`）。
- 同一模块的多个设计文档放在同一子目录下。

### 文档命名
- 格式：`编号.子编号 设计主题.md`（如 `01.02. 任务创建.md`、`01.02.03 标注配置.md`）。
- 独立的业务设计可放在 `99 业务设计/` 子目录。

### 特殊目录
- `00 备份/`：存放废弃或被替代的旧版设计，保留历史参考但不作为当前实现依据。
- `99 参考文档/`：外部参考资料，不作为本项目设计约束。

## 文档膨胀控制

- 单个设计文档建议不超过 **300 行**（含代码示例和表格）。
- 超过时应拆分为多个子文档，在父文档中通过引用链接组织。
- 禁止在文档末尾以日期为标题不断追加独立变更；增量变更应融入对应章节。

## 变更纪律

- 代码实现前**必须**先确认设计文档是否覆盖本次变更；未覆盖的需先补充设计。
- 修改设计文档时，检查是否需要同步更新对应的测试设计文档（03-tr）。
- 设计决策发生变更时，在文档对应章节直接修改，不保留"变更历史"段落（AGENTS.md 的版本历史除外）。
