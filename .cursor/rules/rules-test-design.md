# 测试设计与质量保障流程规范

## 核心原则

**所有原始需求在进行开发设计时，必须同步完成测试设计。** 开发设计（02-dr）和测试设计（03-tr）是同一个设计过程的两个产出，不允许先完成开发设计后再"补"测试设计。

## 测试需求与用例管理（03-tr 目录）

### 文档产出时机
- 每当在 `doc/02-dr/` 中创建或修改开发设计文档时，**必须**同步在 `doc/03-tr/` 中创建或更新对应的测试设计文档。
- 测试文档与开发设计文档保持对应关系：开发设计文档的功能模块应能在测试文档中找到对应的测试覆盖。

### 文档内容要求
测试设计文档**至少**包含以下内容：
1. **关联文档**（强制）：用表格列出对应的原始需求(01-or)和开发设计(02-dr)文档路径，建立**需求 → 设计 → 测试**的完整追溯链。
2. **测试范围**：明确本次测试覆盖的功能点、接口、页面。
3. **测试用例**：用表格或列表形式描述用例，每条用例包含：前置条件、操作步骤、预期结果。
4. **自动化测试标识**：标注哪些用例需要自动化、使用什么框架。

### 文档命名
- 与对应开发设计文档保持一致的主题分类前缀，例如：
  - 开发设计：`doc/02-dr/06 预标注/预标注业务流程设计.md`
  - 测试设计：`doc/03-tr/06 预标注/预标注业务流程测试设计.md`

## 自动化测试代码要求

### 通用要求
- 开发实现原始需求的同时，**必须**编写对应的自动化测试代码。
- 测试代码遵循 `rules-test.md` 中的目录映射和命名规范。

### WEB 类系统级测试
- WEB 相关功能（前端页面、用户交互流程）**必须**编写系统级（E2E）测试代码。
- 系统级测试框架统一使用 **Playwright**。
- 测试代码放置于项目的 `test/e2e/` 目录下，内部按功能模块组织子目录。
- 测试应覆盖关键用户流程和页面交互，不要求覆盖所有边界情况，但核心路径必须覆盖。

## Bug 反馈的文档化（91-qa 目录）

### 触发条件
当用户反馈了问题或 Bug 时，**必须**在 `doc/91-qa/` 中形成文档。

### 文档内容要求
91-qa 文档**至少**包含以下章节：
1. **问题描述**：清晰描述问题现象、复现条件。
2. **关联文档**（强制）：列出该问题关联的原始需求（01-or）、开发设计（02-dr）、测试设计（03-tr）文档路径，建立从问题到需求和设计的**双向追溯链**。
3. **根因分析**：说明问题的根本原因，包含必要的代码引用或日志。
4. **涉及文件**：列出修改的文件及修改内容摘要。
5. **设计约束更新**（可选）：若修复涉及设计变更，说明对应设计文档的更新内容。

### 关联文档示例
```markdown
## 关联文档
| 类型 | 文档路径 |
|:---|:---|
| 原始需求 | `doc/01-or/【预标注】预标注采用小模型.md` |
| 开发设计 | `doc/02-dr/06 预标注/预标注业务流程设计.md` |
| 测试设计 | `doc/03-tr/06 预标注/预标注业务流程测试设计.md` |
```

### 文档命名
- 格式：`【主题分类】问题简述.md`（如 `【预标注】标签匹配问题.md`）。

## 执行检查清单

处理需求或 Bug 时，按以下清单自检：

- [ ] 新建/修改了开发设计文档（02-dr）？→ 是否同步创建/更新了测试设计文档（03-tr）？
- [ ] 实现了功能代码？→ 是否编写了对应的自动化测试？
- [ ] 涉及 WEB 页面交互？→ 是否编写了 Playwright E2E 测试？
- [ ] 收到了用户 Bug 反馈？→ 是否在 91-qa 中形成了分析文档？
- [ ] QA 文档中是否填写了关联文档（01-or / 02-dr / 03-tr）？
