# 工具管理规范

## 权限声明
可请示修改

## 1. 角色定位与边界
1. `tools` 用于实现复杂功能，负责业务流程编排与任务收敛。
2. `tools` 原则上只实现业务代码，不重复实现底层通用能力。
3. 底层能力统一通过 `component` 的 API 函数提供，`tools` 通过调用这些 API 完成能力组合。
4. 禁止在 `tools` 中复制 `component` 已有逻辑，避免功能重叠和后续维护分叉。
5. **强制规则**：所有非组件模块（包含 `tools`、`control`、`core` 等）在“查询组件信息”或“调用组件函数”时，必须统一通过 `tools/component_call_tool` 处理，禁止直接导入或直接调用 `component` 代码。
6. `tools/component_call_tool` 是跨模块访问 `component` 的唯一入口；如需新增组件调用能力，应优先扩展该工具，而不是在其他模块重复实现。

## 2. 模块化要求
1. `tools` 必须模块化设计，每个 tool 一个独立目录，例如 `tools/api_client`。
2. 每个 tool 目录建议包含以下文件：
   - `__init__.py`：包初始化文件
   - 主代码文件：按 tool 功能命名
   - `README.md`：该 tool 的说明文档（必须）
   - `requirements.txt`：仅在该 tool 有独立依赖时提供
3. tool 之间应低耦合，跨 tool 复用优先下沉到 `component`。

## 3. README 编写要求（Agent 可判定）
每个 tool 的 `README.md` 必须能让 agent 快速判断“是否应该使用该 tool”，至少包含：
1. 功能描述：该 tool 解决什么问题、适用场景是什么。
2. 输入与输出：关键参数、返回结构、错误返回约定。
3. 依赖组件：明确调用了哪些 `component` API；若不调用组件，需明确标注“本 tool 不直接调用组件 API”。
4. 使用示例：最小可运行示例或伪代码流程。
5. 边界说明：不适用场景、限制条件、可能副作用。

## 4. 实现规范
1. 命名规范：类名使用驼峰，方法和参数使用小写加下划线。
2. 返回约定：统一返回字典，建议包含 `success`、`data`，失败时包含 `error`。
3. 错误处理：使用 `try-except` 捕获可预期异常，并返回可诊断信息。
4. 日志要求：记录关键业务节点与失败原因，便于追踪问题。

## 5. 依赖与发布
1. 依赖项在对应 `requirements.txt` 中声明；无独立依赖可不单独维护。
2. 发布前需补齐最小测试与 README 更新，保证可读、可用、可维护。
