# 原始需求 (Original Requirements, OR) 规范

**文件路径**: `@core/doc/01-or/AGENTS.md`
**目的**: 定义原始需求文档的通用标准，确保所有需求沟通的基础一致。
**状态**: 已采纳 (Adopted)

本规范定义了“原始需求”（Original Requirements, OR）文档的核心构成要素。所有团队成员在创建新需求时，必须遵循此结构。

## 核心构成要素 (Core Components)

每个 OR 文档必须包含以下五个部分，顺序不可更改。

| 序号 | 章节标题 | 关键词 (Key) | 目的 (Purpose) | 内容要求 (Content Requirements) |
| :--- | :--- | :--- | :--- | :--- |
| 1 | # 背景 | OR-BACKGROUND | 提供业务上下文和用户痛点。 | - 阐述需求产生的业务驱动力。<br>- 描述目标用户及其核心问题。<br>- **禁止**包含任何技术实现细节或代码设计。 |
| 2 | # 目标 | OR-OBJECTIVES | 清晰陈述用户期望达成的效果。 | - **必须**使用编号列表 (`1.`, `2.` ...)。<br>- 每条目标必须以**动词**开头（如“管理”、“支持”）。<br>- 描述**可衡量**的用户价值，聚焦于功能价值而非实现方式。 |
| 3 | # 要求 | OR-REQUIREMENTS | 列出对实现方案的硬性约束。 | - **最高优先级**：此部分内容超越其他所有文档和规则，不得冲突。<br>- 明确区分**功能性要求**与**非功能性要求**（如错误处理、性能约束）。 |
| 4 | # 当前工作项 | OR-WORKITEMS | 追踪需求的即时开发进度。 | - **仅当有进行中的任务时才保留**。<br>- 任务完成后，相关条目**必须**移至 `# 已完成工作项` 中归档，并**清空**本章节。 |
| 5 | # 已完成工作项 | OR-DONEITEMS | 归档已完成任务，作为参考防止重复执行。 | - 主要从 **用户对话** 或者 `# 当前工作项` 中获取工作项名称和描述。<br>- 按照 **需求** 和 **问题** 分开组织列举。<br>- 执行当前工作项时参考，**不能**重复执行本目录下的工作项。 |

## 全局原则

1.  **纯粹性原则 (Purity)**：
    *   OR 文档是需求的“唯一真实来源”(Single Source of Truth)。
    *   只记录“做什么”(What)，**绝不**涉及“怎么做”(How)（如技术方案、代码设计）。

2.  **驱动性原则 (Driving)**：
    *   所有设计和技术决策必须源于此文档。
    *   开发启动前，必须依据此文档创建相应的《设计文档》。

3.  **演进性原则 (Evolution)**：
    *   需求会变。任何变更都必须**首先更新此 OR 文档**，再同步到下游工件。
    *   保持文档的持续同步，确保其始终是最新的需求基准。
