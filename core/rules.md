# 最高原则
1. 任意目录下面的 rules.md 不要修改，只能读取
2. 所有文档或者代码 "权限声明"
- 不允许修改：只能读取，不要修改
- 可请示修改：修改前，需要有 "修改建议" 和 "修改对比"，需要得到人工确认；人工确认可以统一或者拒绝
- 可自行修改：不需要请示，可自行修改

# 工程管理
1. 每个工程独立的目录，例如当前目录 "core"
2. 工程下必须包含如下目录，作用分别如下
- code 代码目录，所有代码都需要放到这个目录下，可以包含部分代码分析和说明文档
- doc 文档，所有文档放到这个目录下
- tools 工具
- rules.md 规则，子目录下也可以定义rules.md，如果多个 rules.md有冲突，子目录下的相关处理依据层级最接近它的 rules.md
3. code 和 tools工具的划分原则
- 与外界交互的代码尽量放到 tools 中
- code 尽量聚焦内部处理，包括本地数据存储与内部的业务逻辑处理

# 文档管理（doc）
1. 每个工程的需求、设计和问题等文档全部放到 doc 目录下
2. doc 目录下文档名称和含义如下
- 01-or 用户原始需求文档，普通用户能够理解的语言，不要有太专业的描述
  - 固定章节为 背景、 目标、 要求、 当前工作项（已实现和已完成直接忽略）
  - 不要在 or 中进行详细设计，可以在上述章节中更新完善一些 OR 描述
  - or 规则和范例参见 "rules/rules-or.md"
- 02-dr 开发设计文档，可以包括前后端以及数据库设计
- 03-tr 测试需求文档
- 44-bak 删除的备份文档，一般不要直接删除， 放进来时如果重名，名称可以添加日期时间戳
- 91-qa 测试问题相关文档，提交的问题揭露方到该目录下
- 99-ops 运维文档，用户说明文档，发布文档都可以放这里面
  - requirements.txt：Python依赖文档
  - release-note.md： 发布文档，需要包含版本号、发布时间、新增功能、变更功能、解决问题和遗留问题等内容

# 软件开发过程管理
1. 整体按照如下顺序：分析需求(或者分析问题) > 撰写或者更新设计文档 > 实现代码 > 发布版本
2. 按照如下顺序梳理文档，01-or > 02-dr > 03-tr
3. 每次完整的需求都需要充分测试，并发布一个版本

# 软件架构
1. 后端使用 Django 的 ORM 来管理数据库，但数据库操作尽量使用原生 SQL 语句
2. 后端（Python）引用新的组件时，更新 requirements.txt 文档
3. 前端不用引入复杂的框架，使用 Django 及其 模板功能即可，全 Python 框架，方便调试
