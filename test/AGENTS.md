# test 目录说明

`test` 目录用于维护项目测试代码，目标是让测试结构与被测代码结构保持稳定映射，降低定位与维护成本。

## 目录用途

- 存放单元测试、集成测试等自动化测试代码。
- 通过目录映射快速定位“某段业务代码对应的测试位置”。

## 文档与代码范围

- 允许内容：测试代码、测试夹具、测试数据、测试辅助工具。
- 不建议内容：与测试无关的业务实现代码、临时脚本长期滞留。

## 目录命名与映射规则（必须遵循）

- 测试目录按被测目录层级镜像组织。
- 每一级测试目录名使用 `test_` + 被测试目录名。
- 映射规则：被测路径 `a/b/c` 对应测试路径 `test/test_a/test_b/test_c`。

示例：
- 被测目录：`component/observe/screen_capture_component`
- 测试目录：`test/test_component/test_observe/test_screen_capture_component`

## 文件命名与组织

- 测试文件命名统一使用 `test_*.py`。
- 测试文件放在对应映射目录下，避免跨模块混放。
- 新增被测模块时，同步补齐对应测试目录与基础测试文件。

## 变更流程

- 当被测目录结构变更时，先更新 `test` 目录映射，再调整测试实现。
- 测试规则变更优先更新 `.cursor/rules/rules-test.md` 与本文件，随后再进行测试代码改造。

## 协作边界

- `test` 仅负责验证行为，不承载业务主实现。
- 与被测目录保持一一映射关系，避免职责交叉与重复实现。
