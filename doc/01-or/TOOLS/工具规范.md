# 背景
当前 `TOOLS` 目录下的需求文档以“单工具能力”为主，缺少对 Cursor 工具体系的统一拆分视角，导致以下问题：
1. “工具边界、调用契约、测试标准、安全约束”常被混写在同一文档中，需求粒度过粗。
2. 同类变更容易反复追加到单一文档，增加理解和追溯成本。
3. 后续同步 `02-dr` 和 `03-tr` 时，难以准确映射到对应设计与测试模块。

# 目标
1. 建立 Cursor 工具需求的解耦结构，按主题拆分为独立 OR 文档。
2. 保证每个子需求文档可独立追溯、独立设计、独立测试。
3. 约束后续 Cursor 工具新增需求按“同主题融入、跨主题新建”的方式维护。

# 要求
## 功能性要求
1. Cursor 工具原始需求必须拆分为以下独立文档，并长期保持：
   - `tool-cursor工具解耦-边界与调用规范.md`
   - `tool-cursor工具解耦-测试与可观测规范.md`
   - `tool-cursor工具解耦-安全与权限规范.md`
2. 后续新增需求时，若属于既有主题（边界/测试/安全），必须融入对应文档章节，不得在本文件堆叠细项。
3. 若需求无法归入既有主题，必须新建独立 OR 文档，并在本文件中补充文档清单。
4. 本文件仅作为 Cursor 工具需求解耦入口，不承载实现细节或测试步骤。

## 非功能性要求
1. 各子文档应保持“单文档单主题”，避免跨主题耦合描述。
2. 每个 OR 文档须满足固定章节顺序，并可直接映射到 `02-dr`、`03-tr` 下对应文档。
3. 文档增量维护时优先修改原章节内容，禁止追加变更日志式段落。
