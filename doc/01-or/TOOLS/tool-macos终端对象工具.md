# 背景
当前 `macos_terminal_tool` 偏“终端对象编排”模式，调用链路包含对象创建、输入输出、读取、关闭等多个步骤。该模式对大模型调用不友好，容易出现参数不全、调用顺序错误、会话状态不一致等问题，导致执行失败率偏高。

本次需求要点是把工具能力收敛为更黑盒的单函数调用模式，降低大模型使用复杂度。

# 目标
1. 建立 `macos_terminal_tool` 的独立原始需求文档，作为工具能力演进唯一需求入口。
2. 将工具对大模型的调用接口简化为“输入一条命令，返回执行结果”的黑盒模型。
3. 降低工具误用概率，避免大模型在多函数和多阶段调用中出错。
4. 保证工具层调用边界清晰：复用组件能力，不重复实现底层终端逻辑。

# 要求
## 功能性要求
1. `tools` 层需提供独立工具目录 `tools/macos_terminal_tool`，并对外暴露一个主调用函数（单函数入口）。
2. 主调用函数的最小输入为一条命令文本（如 `command`），执行后直接返回结果。
3. 主调用函数必须封装内部会话处理细节（创建、执行、回收、关闭），调用方无需感知对象生命周期。
4. 工具调用组件能力时必须统一经由 `tools/component_call_tool.ComponentCallTool.control_call`。
5. 工具返回结构必须统一为：
   - 成功：`success + data`
   - 失败：`success + error`
6. 结果数据至少包含：执行命令、退出码、标准输出（必要时含错误输出）。
7. 组件调用失败、平台不支持、命令执行异常等场景必须返回可读错误。
8. 对外不再要求大模型直接调用对象化接口（如创建对象、读取增量输出、关闭对象等多步骤 API）。

## 非功能性要求
1. 工具接口命名与参数设计应优先面向大模型可调用性，避免多阶段状态依赖。
2. 工具不重复实现 shell 子进程管理，保持“工具编排、组件执行”的职责边界。
3. 工具需明确平台边界：当前版本支持系统为 `macos`。
4. 文档、设计、测试需保持追溯一致，后续变更必须同步到 `02-dr/03-tr`。
5. 工具需具备最小可运行 demo，便于快速验证基础可用性。
