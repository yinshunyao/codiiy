# 背景
1. 会话处理相关原始需求目前分散在多处，前后端边界不够清晰，导致沟通与实现口径不一致。
2. 现阶段需要将“会话处理”统一沉淀为单一需求文档，明确前端与后端各自职责，减少重复确认成本。
3. 本次整理仅针对会话处理链路；自主规划与智能体相关需求沿用现有定义，不在本次调整范围内。

# 目标
1. 统一会话处理需求口径，沉淀前端与后端的职责边界。
2. 支持会话从输入、展示、回退到存储的完整闭环，并保持可追溯。
3. 支持文本、语音与附件在会话中的一致处理，保证多模态场景可用。
4. 保持自主规划与智能体能力定义不变，避免对既有能力链路造成回归。

# 要求
## 功能性要求
1. 会话处理需求必须按前端与后端拆分管理，且以本文件作为统一来源。
2. 前端会话处理要求：
   - 支持会话列表与消息内容展示，消息按会话维度组织并可追溯。
   - 支持用户发送文本、语音和附件消息，语音消息需支持识别文本的折叠与展开查看。
   - 支持对指定用户消息执行回退，回退后需恢复消息文本与关联附件到输入区，便于二次编辑发送。
   - 支持多项目下的会话切换与展示，当前会话必须明确归属当前项目。
   - 支持会话删除操作，并在界面反馈删除结果。
3. 后端会话处理要求：
   - 会话原始记录必须持久化存储，数据库首期支持 SQLite。
   - 每次对话必须记录上下文，并按会话连续关联，保证后续整理可基于完整历史。
   - 会话处理需遵循需求澄清规则：信息完整时返回确认语义；信息不完整时返回澄清问题。
   - 会话消息需支持多模态数据入库与读取，保证与前端输入能力一致。
   - 会话必须按项目归属隔离存储与查询，不同项目间不得混写。
4. 前后端协同要求：
   - 会话展示内容必须以数据库中的会话记录为准，前后端字段语义保持一致。
   - 语音识别结果需参与会话处理与存储，并可在前端按消息维度查看。
   - 会话回退能力需前后端共同支持，确保“回退-编辑-重发”链路可用。
   - 用户对话中“智能体规划/任务处理实时反馈”的展示链路需求，统一以 `doc/01-or/【智能对话】智能体规划和回复的前后端处理.md` 为准，本文件不重复展开。
5. 自主规划与智能体要求保持不变：
   - 继续沿用现有自主规划策略与执行约束，不在本需求中新增或删减。
   - 继续沿用现有智能体能力边界、调用约束与协同机制，不在本需求中调整。

## 非功能性要求
1. 优先复用 Django 现有能力，不重复建设基础框架能力。
2. 会话链路需保证可观测与可追溯，关键状态与处理结果可定位。
3. 会话数据读写需保证一致性，避免出现前端显示与后端存储不一致。
4. 需求描述保持原始需求层级，不引入实现细节、代码方案或接口设计细节。
