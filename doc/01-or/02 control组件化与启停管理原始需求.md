# 背景
当前 `component`（组件）目录已具备字符串路径调用能力，但功能代码在目录组织上仍偏“模块聚合”，组件边界不够清晰。  
随着 `observe`、`handle`、`communicate`、`decide` 等能力持续扩展，现有方式在新增组件、升级组件、灰度替换、故障隔离方面成本逐步升高。

此外，现阶段缺少“组件启用/停用”统一机制，无法对问题组件进行快速隔离，也不利于按项目或环境控制组件可用性。

# 目标
1. 规范 `component` 及子目录的组件化目录标准，支持每个组件以独立文件夹组织和维护。
2. 支持组件平滑升级，兼容历史调用路径，避免升级导致已接入方调用中断。
3. 提供组件启用和停用能力，支持运行时按组件控制可调用状态。
4. 保持现有对外调用方式可用，降低改造接入成本。

# 要求
## 功能性要求
1. `component` 下能力目录（如 `observe`、`handle`、`communicate`、`decide`）需支持“组件目录标准化”，每个组件为独立文件夹。
2. 组件目录层级需要简化，不使用 `components` 中间层，组件目录直接放在模块目录下（如 `component/observe/<component_name>`）。
3. 组件目录应可独立新增与独立升级，不影响其他组件目录职责边界。
4. 每个组件目录需内聚维护本组件的依赖与说明，至少包含组件级 `requirements.txt` 与组件自述文档。
5. 组件共享实现（如模型客户端）应归属到具体组件目录，避免长期滞留在模块外层散落文件。
6. 系统需提供组件注册信息管理能力，至少包含组件标识、所属模块、导出函数路径、默认启用状态。
7. 系统需提供组件启用/停用管理能力：
   - 支持按组件启用或停用；
   - 停用后，组件导出函数不可被调用，并返回明确错误信息；
   - 启用后恢复可调用。
8. 系统需提供升级兼容策略：
   - 兼容策略需明确且可审计，不依赖代码中的隐式转发层；
   - 对外仅允许使用文档登记的规范函数路径，不提供未登记旧路径兜底；
   - 升级变更需通过文档与索引配置显式发布，避免“看似可用但规则不清”的隐式兼容。
9. `component.call_by_path` 及相关工具调用流程需遵循组件启停与兼容映射规则。
10. 在 `component/communicate` 下新增三方通信组件能力，至少包含：
   - 钉钉通信组件；
   - 企业微信通信组件；
   - 飞书通信组件；
   且每种通信方式必须位于独立组件目录，解耦实现与维护。
11. 通信组件需支持“运行时显式传参”与“按配置标识读取配置”两种参数来源，避免强依赖本地明文配置文件。
12. 系统需支持通过 Django 管理界面维护通信组件配置项，并存储到数据库；组件按规则（provider + 配置名）读取并使用对应参数。
13. 组件必须声明待配置参数清单（如 webhook_url、app_key、app_secret 等），并在模块文档中可被平台读取和展示。
14. 需要形成通用机制：组件中“系统参数名”和“参数值”均可动态扩展，不预设固定字段集合。
15. 组件需在自身规则中声明参数配置 Schema（字段名、必填、敏感、说明、默认值等），并随组件代码同仓维护。
16. Django 在“组件功能查询页”读取组件规则后，若检测到存在系统参数 Schema，需展示“参数配置”按钮，支持用户新增/编辑该组件参数配置。
17. 参数配置存储需按“组件标识 + 配置名”管理，支持同一组件维护多套配置（如 test/prod）。
18. 组件功能管理页的上传下载粒度需基于“单个组件”而非“整个模块目录”：
   - 下载时仅打包目标组件目录（如 `component/communicate/dingtalk_component`）；
   - 上传时仅允许解压覆盖目标组件目录，不得写入同模块下其他组件目录；
   - 上传解压需保留路径安全校验，禁止 zip slip（目录穿越）写入目标目录外路径。
19. 组件功能管理页上传入口需统一为一个“上传”按钮，放在搜索按钮旁边；上传后由服务端解压并在当前模块下新建目标组件目录。
20. 函数卡片中的组件操作入口保留“参数配置”和“下载组件”，不再在每条函数卡片内重复提供上传入口。
21. 部分组件 API 测试耗时较长，API 测试页需支持“异步执行测试任务”：
   - 点击“执行测试”后立即创建测试任务并返回任务状态，不阻塞页面请求；
   - 在任务执行完成前，测试按钮置为不可点击，并展示“正在测试中”状态提示；
   - 任务完成后自动恢复按钮可用，并展示本次入参与测试结果（成功/失败）。

## 非功能性要求
1. 组件状态管理应简洁可追踪，便于问题定位与运维操作。
2. 升级兼容策略应尽量减少对现有调用方的改造量。
3. 文档与实现需保持一致，组件新增、升级、停用后可在目录自述信息中体现。
4. 在不改变现有业务语义前提下完成改造，避免大范围行为偏移。
