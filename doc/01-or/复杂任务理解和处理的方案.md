# 背景
当前复杂任务的执行效果不稳定，主要问题集中在两类：
1. 复杂任务时，`mindforge` 执行步数和上下文预算偏保守，容易在尚未完成关键动作前提前结束。
2. 工具调用链路中，规划阶段虽然会生成候选 fallback 工具，但执行阶段未充分利用，导致失败后“可重试但未重试”。

现有代码已经具备 `auto/react/plan_execute/reflexion/cot` 策略与授权校验能力，本次优化应在既有架构内增强，不引入偏离当前实现的大重构。

# 目标
1. 提升复杂任务成功率：复杂查询在多步执行场景下，减少“达到最大步数限制”导致的失败。
2. 提升工具调用智能度：当主工具调用失败且存在同域候选工具时，支持自动回退重试。
3. 保持安全边界：授权失败、白名单拒绝等不可恢复错误不得盲目重试。
4. 保持可观测性：工具调用链路需保留主目标、回退目标、尝试次数、是否触发回退等信息。
5. 降低回归风险：兼容现有策略路由、授权逻辑与编排输出结构。

# 要求
## 功能性要求
1. `mindforge_runner` 需按任务复杂度动态调整 `ReActEngineConfig`（至少包含 `max_steps` 与 `max_tokens`），复杂任务预算高于简单任务。
2. `planner` 在校验 LLM 计划时，`tool` 步骤生成 `fallback_targets` 时必须纳入候选工具集，而非固定空列表。
3. `coordinator` 执行 `tool` 步骤时，需按“主目标 + fallback_targets”顺序尝试，最大尝试次数可配置且默认大于 1。
4. 工具重试需包含停止条件：当错误属于授权、白名单或参数结构硬失败时，立即停止后续 fallback 尝试。
5. 工具执行结果结构中需明确返回：
   - `selected_function_path`（最终选中工具）
   - `attempt_count`（尝试次数）
   - `fallback_used`（是否使用回退工具）
   - `attempt_trace`（每次尝试明细）
6. 现有失败后 `mindforge replan` 机制保留，但应在“本地 fallback 尝试耗尽”后触发，避免过早进入重规划。
7. `auto` 策略需区分“执行型任务”与“解释型任务”：当任务目标包含执行动作且存在可执行工具链时，必须优先走可执行策略链路（`react/plan_execute/reflexion`），不得以纯文本策略直接收敛。
8. 对执行型任务，`auto` 在阶段结果判定时需满足“执行闭环”条件：至少发生一次有效工具调用（非空工具名且调用未报错）；仅返回“下一步建议/可执行方向”不得判定为任务完成。
9. 对执行型任务，若可执行策略链路全部失败，应返回明确失败与阻塞原因，禁止通过 `cot` 兜底输出建议文本并标记成功，避免“方向正确但未执行”的假成功。
10. 对执行型任务，`auto` 在每个可执行阶段内需支持“替代方案重试”：首轮失败或执行闭环不足时，在不重复同一路径的前提下至少再尝试一次替代思路（如改用不同工具组合、调整步骤拆分），不得因单次失败提前结束。
11. `auto` 仅在以下条件之一满足时结束：任务已达成且满足执行闭环；或全部阶段及其替代重试均耗尽。其余情况必须继续尝试剩余可选策略路径。
12. `auto` 模式进入执行前必须先进行“预热判定”：向 LLM 提供完整可用工具清单（工具名称 + 简要功能 + 调用路径），并要求返回是否已可直接回答（`is_completed`）。
13. 预热判定若返回已完成，`auto` 需直接返回最终答复并结束；若未完成，必须依据预热建议的下一策略（`cot/react/plan_execute/reflexion`）继续执行。
14. 当下一策略为 `react/plan_execute` 时，提示词需显式携带工具名称、功能描述与 API 调用方式（`{"tool":"...","kwargs":{...}}`），并支持将多命令任务拆分为按顺序逐条执行的步骤。
15. `auto` 在执行阶段应优先沿预热建议路径推进，避免无依据地随机遍历工具；仅在失败或约束不满足时进行有限替代重试。

## 非功能性要求
1. 不改变现有授权边界与白名单原则。
2. 不新增与本需求无关的组件或目录。
3. 设计与实现保持小步可回滚，优先修改现有模块：`planner`、`coordinator`、`mindforge_runner`。
4. 测试需覆盖回退重试、停止重试和动态预算核心路径。

# 已完成工作项
**需求**
- 修复 ReAct 工具调用结果丢失：当工具返回 `{"success":true,"data":...}`（无 `data.result`）时，执行器仍应正确写入 observation，不得出现 `null`。
- 修复 ReAct 工具名到函数路径映射缺口：`mindforge_runner` 在 query 检索未命中时仍补齐白名单工具集函数，避免 `macos_terminal_tool` 这类工具名出现“已授权但未注册”。
- 收敛工具函数自动发现边界：代码反射仅提取工具主类公共 API，避免内部辅助类方法进入候选导致别名歧义。
- 工具能力索引解耦：`capability_search` 不再依赖固定 `_TOOL_FUNCTION_CATALOG`，改为从工具目录 `README.md` API 与代码反射自动发现可用函数。
- 统一工具能力来源：前端工具查询与自主规划候选工具共享同一动态索引来源，新增工具目录后无需手工补清单。
- 修正总结后重规划触发门控：`completion_signal.need_replan=true` 时必须触发重规划执行，`has_alternative` 仅作提示不作拦截。
- 收敛并放宽复杂任务执行预算：按复杂度动态提升 `mindforge_runner` 的 `max_steps/max_tokens`，降低关键动作前提前结束概率。
- 补齐工具回退执行约束：`coordinator` 支持可配置 `tool_max_attempts`，默认大于 1，且仅在本地 fallback 尝试耗尽后触发 replan。
- 强化不可恢复错误停止条件：授权/白名单/参数结构硬失败命中后立即停止 fallback，避免无效重试。
- 增强工具步骤可观测输出：统一返回 `selected_function_path`、`attempt_count`、`fallback_used`、`attempt_trace` 供轨迹与汇总消费。
- 补充回归测试覆盖动态预算、回退上限配置与参数硬失败停止路径。
- 优化 `auto` 预热判定链路：预热阶段注入完整工具清单并返回 `is_completed`，可直接结束或继续后续策略。
- 新增 `auto` 预热驱动的策略执行：未完成时按预热建议继续；`cot` 走文本推理，`react/plan_execute` 携带工具 API 调用方式并按执行目标推进。
