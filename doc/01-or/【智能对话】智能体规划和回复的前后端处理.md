# 背景
1. 用户在聊天中发起问题后，智能体已经具备“自主规划 + 任务处理 + 最终回复”的完整执行能力。
2. 当前相关要求分散在多份文档中，前后端展示口径存在重复与交叉，维护成本较高。
3. 需要将“对话过程信息如何展示、如何实时反馈”的前后端协同需求统一收敛到本文件，作为单一来源。

# 范围与边界
1. 本文档仅定义“前后端信息展示与状态反馈逻辑”。
2. 智能体算法策略、心法路由、业务处理规则、工具执行逻辑保持不变，仍以既有原始需求文档为准。
3. 本文档不新增算法能力，不调整业务执行链路，只约束用户可见信息与状态同步行为。

# 目标
1. 统一用户对话中“处理中间步骤 + 最终回复”的展示口径。
2. 统一前后端在任务状态、步骤轨迹、消息收敛上的职责边界。
3. 保证用户在任务执行中可持续感知进度，在任务结束后看到一致且可读的最终状态。

# 要求
## 功能性要求
1. 对话执行阶段展示
   - 用户发送消息后，前端需进入任务执行态，并展示可感知的处理中状态。
   - 执行过程至少区分“规划相关步骤”“工具/任务处理步骤”“结果收敛步骤”三类可读轨迹。
   - 过程轨迹需支持实时更新，避免用户只能看到静态占位文案。

2. 任务状态与过程轨迹
   - 后端需提供可持续拉取的任务状态快照（含任务状态、过程步骤、最终结果摘要）。
   - 任务状态至少支持：`pending`、`running`、`success`、`failed`、`stopped`。
   - 过程步骤状态需闭环：进入 `running` 的步骤，最终必须收敛到 `success/failed/stopped` 之一。
   - 前端展示以服务端返回状态为准，不得依赖仅前端推断状态。

3. 过程步骤信息展示
   - 每个步骤至少包含：步骤类型、步骤标题或说明、执行状态、时间戳。
   - 大模型调用步骤可展示 token 信息（输入、输出、总计）；上游无数据时需展示“无统计数据”，不得阻断主流程。
   - 展示内容需以用户可读文本为主，禁止将底层 SDK 原始对象字符串直接展示给最终用户。

4. 最终回复收敛
   - 任务进入终态后，前端需展示与终态一致的助手消息，不得长期停留“思考中”等占位词。
   - 当任务失败或停止时，后端需回写可读终态文案，前端据此更新展示。
   - 最终回复与过程轨迹在同一轮任务中需可追溯关联，避免“有轨迹无结果”或“有结果无轨迹”。

5. 刷新恢复与持续反馈
   - 页面刷新后，如存在有效 `pending/running` 任务，前端需基于后端快照自动恢复状态拉取。
   - 任务无效或已终态时，前端不得继续显示执行中态，避免误导用户。
   - 轮询或流式更新期间，前端不应强制打断用户当前查看状态（如滚动位置、已展开步骤）。

6. 前后端协同一致性
   - 过程轨迹、最终回复、任务状态三者字段语义需保持一致，避免同一任务多口径冲突。
   - 聊天页面展示内容必须可由后端持久化数据与任务快照回放，不依赖临时内存态才能成立。

7. 过程步骤树状结构展示
   - 前端过程步骤展示需支持树状结构，降低长链路任务的阅读成本。
   - 对“有明确开始与结束”的工具调用或流程调用，前端应聚合为一个父步骤节点，节点内同时展示开始信息与结束信息，避免拆成大量离散条目。
   - 当某流程内部触发子流程（如触发 ReACT、Reflexion 或其他规划流程）时，需以子节点挂载在对应父流程节点下，体现调用层级关系。
   - ReACT、Reflexion 等响应式分步执行过程应按步骤逐项展示，支持并行分支结构的可视化表达与状态区分。
   - 树状节点状态需与任务状态体系一致，并支持父子节点状态联动（子节点未收敛时父节点不得提前标记完成）。

## 非功能性要求
1. 展示逻辑需具备可观测性，问题发生时可基于任务状态与步骤轨迹定位。
2. 文档描述保持需求层，不展开具体接口实现或代码方案。
3. 后续如新增展示字段，应在本文件追加，避免回流到算法或业务文档造成重复。
