# 背景
当前工程需要同时支持「核心工程（core）」与「多项目衍生工程（projects/*）」并行管理。  
在原始需求收集、会话处理、角色任务管理、工具扩展等场景中，目录职责边界不清会导致以下问题：
- 新增项目时目录复制不完整，后续运行与维护成本增加；
- 会话、规则、工具、文档等资产分散，难以定位和复用；
- 不同项目之间结构不一致，造成功能对齐和迁移困难。
此外，`component`、`handle`、`observe`、`decide` 等目录下的功能函数若长期以“散落文件”方式维护，会导致组件新增慢、升级风险高、复用困难。

因此需要在现有初步规划基础上，统一项目整体结构定义，作为后续设计与实现的基础约束。

# 目标
1. 明确工程根目录下各一级目录的职责边界，保证团队在新增内容时有统一落点。
2. 规范 `projects` 下衍生项目的标准骨架，支持多项目快速创建与持续维护。
3. 约束组件层、角色、任务、工具、文档、数据存储等模块的关系，减少重复建设与目录漂移。
4. 支持后续功能扩展时保持结构稳定，确保不同项目间具备可对齐、可迁移、可复用能力。
5. 支持组件层能力组件化管理，使功能组件可以独立新增与升级，降低维护成本。

# 要求
## 功能性要求
1. 根目录固定包含以下核心目录：`component`、`core`、`doc`、`data`、`projects`、`tools`。
2. `component`（组件）目录用于对外能力编排，至少包含：
   - `component/communicate`：沟通相关能力；
   - `component/handle`：执行与处理相关能力；
   - `component/observe`：观察与感知相关能力；
   - `component/decide`：决策相关能力。
3. `component` 及其子目录（如 `handle`、`observe`、`communicate`、`decide`）下的功能函数应按“组件”进行组织，每个组件使用独立文件夹管理。
4. 组件目录需支持独立扩展，新增组件时不影响现有组件的目录边界和职责。
5. 组件目录需支持独立升级，组件升级应尽量控制影响范围，避免对非目标组件造成连带改动。
6. `projects` 目录用于管理衍生项目，每个项目（如 `proA`、`proB`）应采用统一骨架，至少包含：
   - `component`
   - `core`
   - `doc`
   - `data`
   - `tools`
7. `data/roles` 目录用于角色定义与约束，`tools` 目录用于与外部交互或复用工具能力。
8. `doc` 目录统一承载需求、设计、测试、运维等文档。
9. `data` 目录统一承载运行数据与临时数据，至少包含：
   - `data/cache`：缓存目录；
   - `data/database`：数据库文件目录；
   - `data/roles`：角色定义与约束目录（由原顶层 `roles` 下沉）；
   - `data/temp`：临时文件或临时信息目录；
   - `data/tasks`：任务数据目录（由原顶层 `tasks` 下沉）。
10. 顶层不再保留 `memory`、`roles`、`tasks`、`temp` 目录，统一纳入 `data` 管理。
11. 新建衍生项目时，需按标准骨架初始化，确保关键目录完整可用。
12. 管理端前端界面中的组件模块导航需与 `component` 目录定义保持一致，至少支持 `observe`、`handle`、`communicate`、`decide` 的展示与切换，并具备后续新增模块时的可扩展能力。
13. 组件功能列表页的工具栏操作需保持紧凑一致：搜索输入框宽度控制为页面可用宽度的约 50%，搜索与上传操作在同一行连续展示，上传入口支持点击后直接选择 `.zip` 并触发上传，由服务端完成解压处理；上传下载粒度需基于单个组件目录，不再基于整个模块目录。
14. 管理端前端界面需同时支持暗色与亮色两套主题风格，并在左侧栏底部用户区提供开关式主题切换入口；切换结果应在各页面保持一致，避免仅单页生效。
15. 左侧栏底部用户区不再直接展示退出按钮；点击用户名或头像进入个人设置页面，在个人设置页面内提供退出按钮、主题切换与修改密码能力。
16. 左侧“设置”分组的二级菜单不再提供“修改密码”入口，避免与个人设置中的账号能力入口重复。
17. 管理端左侧导航增加一级菜单“模型”，并包含两个固定二级菜单：
   - 大模型API：用于配置云端大模型调用参数；
   - 本地大模型：用于配置和管理本地运行模型。
18. 大模型API页面需支持配置至少以下信息：
   - 配置名称；
   - 厂商名称；
   - API Base URL；
   - API Key（可编辑与保存）；
   - 默认模型ID；
   - 启用状态。
19. 大模型API配置需支持列表、创建、编辑、删除等基础管理能力。
20. 本地大模型页面需支持配置至少以下信息：
   - 配置名称；
   - Ollama 服务地址；
   - 模型名称；
   - keep_alive；
   - 启用状态。
21. 本地大模型页面需提供运行管理能力，至少包含：
   - 查询本地模型列表；
   - 查询当前运行中的模型；
   - 拉取模型；
   - 加载（预热）模型；
   - 卸载模型。
22. 本地大模型运行管理默认以 Ollama 官方 HTTP 接口为基础，服务不可用时需返回可读错误提示，不允许页面直接报错中断。
23. 大模型API配置页面需内置主流云厂商预设，至少包含阿里、火山、腾讯、华为的常用配置项。
24. 大模型API配置页面表单顶部需提供“厂商预设下拉选择”，用户选择后自动填充配置名称、厂商名称、API Base URL、默认模型ID等字段。
25. 大模型API配置页面需做基础视觉优化，保证预设选择区、表单区和提示信息层次清晰，操作路径简短。
26. 本地大模型配置页面需提供“本地模型预设下拉选择”，至少覆盖常用 Ollama 模型与参数版本组合。
27. 选择本地模型预设后，需自动填充配置名称、Ollama 地址、模型名称、keep_alive 等基础字段，并允许用户再手动调整。
28. 聊天对话页需展示大模型 token 计数信息：
   - 在“处理中间步骤”中，对每次大模型调用展示 token 统计（至少包含输入、输出、总计）；
   - 在“最终回复结果”中展示本次回复累计 token 统计（至少包含输入、输出、总计）；
   - 当上游模型未返回 token 数据时，页面需保持可读展示并明确为“无统计数据”，不影响回复主流程。
29. 聊天对话页的“处理中间步骤”需明确标识步骤类型，至少区分：
   - 大模型调用步骤（如规划、汇总、回复生成）；
   - 工具/代码调用步骤（组件或命令执行）；
   - 普通流程步骤（状态更新、结果写回）。
30. 页面展示给用户的回复文本需做可读化处理：
   - 禁止直接展示底层 SDK 原始对象/字典字符串（如 `{'success':..., 'data':...}`）；
   - 当模型返回结构化包裹结果时，优先提取自然语言正文（`text/content`）展示；
   - 仅在调试轨迹区保留必要结构化字段，且需避免将原始对象表达式直接呈现给最终用户。

## 非功能性要求
1. 目录命名需保持清晰、稳定、可读，避免同义重复目录并存。
2. 结构调整应优先兼容已存在代码与文档，避免大幅偏离既有实现。
3. 项目结构应具备可扩展性，允许在不破坏一级职责边界的前提下逐步细化子目录。
4. 文档内容聚焦结构与职责定义，不展开具体实现细节。
5. 组件化目录应保证命名和层级稳定，便于长期迭代中的组件替换与版本演进。
