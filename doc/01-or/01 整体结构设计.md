# 背景
当前工程需要同时支持「核心工程（core）」与「多项目衍生工程（projects/*）」并行管理。  
在原始需求收集、会话处理、角色任务管理、工具扩展等场景中，目录职责边界不清会导致以下问题：
- 新增项目时目录复制不完整，后续运行与维护成本增加；
- 会话、规则、工具、文档等资产分散，难以定位和复用；
- 不同项目之间结构不一致，造成功能对齐和迁移困难。
此外，`component`、`handle`、`observe`、`decide` 等目录下的功能函数若长期以“散落文件”方式维护，会导致组件新增慢、升级风险高、复用困难。

因此需要在现有初步规划基础上，统一项目整体结构定义，作为后续设计与实现的基础约束。

# 目标
1. 明确工程根目录下各一级目录的职责边界，保证团队在新增内容时有统一落点。
2. 规范 `projects` 下衍生项目的标准骨架，支持多项目快速创建与持续维护。
3. 约束组件层、角色、任务、工具、文档、数据存储等模块的关系，减少重复建设与目录漂移。
4. 支持后续功能扩展时保持结构稳定，确保不同项目间具备可对齐、可迁移、可复用能力。
5. 支持组件层能力组件化管理，使功能组件可以独立新增与升级，降低维护成本。

# 要求
## 功能性要求
1. 根目录固定包含以下核心目录：`component`、`core`、`doc`、`data`、`projects`、`tools`。
2. `component`（组件）目录用于对外能力编排，至少包含：
   - `component/communicate`：沟通相关能力；
   - `component/handle`：执行与处理相关能力；
   - `component/observe`：观察与感知相关能力；
   - `component/decide`：决策相关能力。
3. `component` 及其子目录（如 `handle`、`observe`、`communicate`、`decide`）下的功能函数应按“组件”进行组织，每个组件使用独立文件夹管理。
4. 组件目录需支持独立扩展，新增组件时不影响现有组件的目录边界和职责。
5. 组件目录需支持独立升级，组件升级应尽量控制影响范围，避免对非目标组件造成连带改动。
6. `projects` 目录用于管理衍生项目，每个项目（如 `proA`、`proB`）应采用统一骨架，至少包含：
   - `component`
   - `core`
   - `doc`
   - `data`
   - `tools`
7. `data/roles` 目录用于角色定义与约束，`tools` 目录用于与外部交互或复用工具能力。
8. `doc` 目录统一承载需求、设计、测试、运维等文档。
9. `data` 目录统一承载运行数据与临时数据，至少包含：
   - `data/cache`：缓存目录；
   - `data/database`：数据库文件目录；
   - `data/roles`：角色定义与约束目录（由原顶层 `roles` 下沉）；
   - `data/temp`：临时文件或临时信息目录；
   - `data/tasks`：任务数据目录（由原顶层 `tasks` 下沉）。
10. 顶层不再保留 `memory`、`roles`、`tasks`、`temp` 目录，统一纳入 `data` 管理。
11. 新建衍生项目时，需按标准骨架初始化，确保关键目录完整可用。
12. 管理端前端界面中的组件模块导航需与 `component` 目录定义保持一致，至少支持 `observe`、`handle`、`communicate`、`decide` 的展示与切换，并具备后续新增模块时的可扩展能力。
13. 智能体、工具集、组件功能列表页的工具栏操作需保持紧凑一致：不再保留各页面独立搜索输入框，上传入口支持点击后直接选择 `.zip` 并触发上传，由服务端完成解压处理；上传下载粒度需基于单个组件目录，不再基于整个模块目录。
14. 管理端前端界面需同时支持暗色与亮色两套主题风格，并在左侧栏底部用户区提供开关式主题切换入口；切换结果应在各页面保持一致，避免仅单页生效。
15. 左侧栏底部用户区不再直接展示退出按钮；点击用户名或头像进入个人设置页面，在个人设置页面内提供退出按钮、主题切换与修改密码能力。
16. 左侧“设置”分组的二级菜单不再提供“修改密码”入口，避免与个人设置中的账号能力入口重复。
17. 管理端左侧导航增加一级菜单“模型”，并包含两个固定二级菜单：
   - 大模型API：用于配置云端大模型调用参数；
   - 本地大模型：用于配置和管理本地运行模型。
18. 大模型API页面需支持配置至少以下信息：
   - 配置名称；
   - 厂商名称；
   - API Base URL；
   - API Key（可编辑与保存）；
   - 默认模型ID；
   - 启用状态。
19. 大模型API配置需支持列表、创建、编辑、删除等基础管理能力。
20. 本地大模型页面需支持配置至少以下信息：
   - 配置名称；
   - Ollama 服务地址；
   - 模型名称；
   - keep_alive；
   - 启用状态。
21. 本地大模型页面需提供运行管理能力，至少包含：
   - 查询本地模型列表；
   - 查询当前运行中的模型；
   - 拉取模型；
   - 加载（预热）模型；
   - 卸载模型。
22. 本地大模型运行管理默认以 Ollama 官方 HTTP 接口为基础，服务不可用时需返回可读错误提示，不允许页面直接报错中断。
23. 大模型API配置页面需内置主流云厂商预设，至少包含阿里、火山、腾讯、华为的常用配置项。
24. 大模型API配置页面表单顶部需提供“厂商预设下拉选择”，用户选择后自动填充配置名称、厂商名称、API Base URL、默认模型ID等字段。
25. 大模型API配置页面需做基础视觉优化，保证预设选择区、表单区和提示信息层次清晰，操作路径简短。
26. 本地大模型配置页面需提供“本地模型预设下拉选择”，至少覆盖常用 Ollama 模型与参数版本组合。
27. 选择本地模型预设后，需自动填充配置名称、Ollama 地址、模型名称、keep_alive 等基础字段，并允许用户再手动调整。
28. 用户对话中的“智能体规划、任务处理、实时反馈、过程步骤与最终回复展示”前后端协同需求，统一收敛至 `doc/01-or/【智能对话】智能体规划和回复的前后端处理.md`，本文件不再重复展开展示细节。
32. `component/handle` 需提供独立“文件写组件”，支持文件写入相关能力，至少包含：
   - 按路径创建文件（必要时自动创建父级目录）；
   - 修改已有文件内容（覆盖写入、追加写入等）；
   - 返回统一结构化结果（成功/失败、关键元信息与可读错误）。
33. 当新增组件或工具后，管理端搜索能力需支持实时更新：
   - 列表可见的新组件/工具，搜索结果中也必须可立即检索；
   - 缓存索引需在新增后自动刷新，避免依赖手工重启或人工清缓存。
34. 程序启动时需执行一次能力搜索索引刷新，确保服务重启后索引与当前组件/工具状态一致。
34.1 组件停用态需在编排全链路生效：停用组件函数不得进入规划候选、不得作为工具回退目标、不得被工具执行器实际调用；当命中停用函数时需返回可读错误并触发替代方案或重规划。
35. 管理端需提供统一“综合搜索”页面，默认同时检索智能体、工具集、组件三类能力，并支持按类型分项筛选（智能体 / 工具集 / 组件）。
36. 左侧菜单下方用户区（个人头像与在线状态区域）需新增搜索入口按钮，点击后进入“综合搜索”页面；该入口需在暗色与亮色主题下均可读、可点击。
37. 自主规划策略需支持解耦扩展：在 `agents/mindforge` 下按“每策略一个独立目录”管理策略实现（至少包含 `react_strategy`、`cot_strategy`、`plan_execute_strategy`、`reflexion_strategy` 与 `auto_strategy`，且各自具备 `README.md`）；执行器保持在原编排目录，不迁移执行入口。
38. 自主规划策略需支持自动路由能力：`auto` 策略需根据任务复杂度、工具依赖和执行阶段选择合适策略（可单策略直达，或按“规划->执行->反思兜底”分层调用），减少重复规划和无效重试。
39. 伙伴聊天对话框在使用自主任务规划策略时，需在顶部模型信息左侧展示“心法模式”；默认心法模式为 `Auto`，并保持与后端实际执行策略一致。
40. 伙伴聊天对话框中的“心法模式”需支持下拉选择并即时生效，用户可在 `auto/react/cot/plan_execute/reflexion` 间切换，切换结果需在后续任务执行中生效并可保持当前伙伴会话配置。
41. 聊天过程步骤中的心法轨迹展示（含 `auto` 子心法可见性）归口到 `doc/01-or/【智能对话】智能体规划和回复的前后端处理.md`，本文件仅保留策略能力与执行边界要求。
42. `auto` 策略需具备工具能力感知：策略选择时不能只看“是否有专用工具”，需基于完整候选工具集判断“是否可通过组合/多步骤工具链完成任务”；存在可执行工具链时优先可执行策略（如 ReAct/Plan-Execute），避免误判到纯文本策略。
43. `auto` 策略需具备防死胡同回退机制：当首选策略失败，或阶段表面成功但结果经达标判定后“不符合任务预期”时，按预设回退链路尝试替代策略并避免重复走同一路径；回退过程需可观测并体现在步骤轨迹中。
44. 编排层 `step_summarize` 不得仅以“有总结文本”判定任务完成；需在总结后增加“任务完成度信号判定”（评估是否已交付目标结果、是否存在可执行替代方案）。当判定为“未执行到位且存在后续方案”时，需自动触发一次重规划执行并在步骤轨迹中可见。
44. `auto` 策略的工具能力感知需升级为“能力标签”机制：基于 `function_path + description` 识别细粒度标签（如目录浏览、文件读写、命令执行、网络请求等），并将标签用于策略路由与可解释展示，提升“可执行任务”识别准确率。
45. 聊天窗口轮询刷新中的用户查看状态保持、刷新后任务恢复轮询、以及展示兜底策略，统一以 `doc/01-or/【智能对话】智能体规划和回复的前后端处理.md` 为准。
46. 聊天对话需支持“单轮交互模式”：当本轮任务执行过程中出现必须由用户补充的信息（如密码、模式选择、确认项）时，助手可在同一轮内发起交互请求并等待用户提交，提交后继续当前轮次执行，不强制用户重发整轮问题。
47. 单轮交互模式至少支持以下交互控件：
   - 密码输入（前端以密码框展示，界面与轨迹中脱敏）；
   - 单项模式选择（下拉或单选，至少支持 value/label 结构）；
   - 普通文本输入（用于补充参数）。
48. 聊天任务状态接口需返回“交互请求快照”（含交互ID、交互类型、标题、可选项、必填约束、提交状态），前端轮询中可即时渲染；提交接口需支持按任务ID + 交互ID回填用户输入并驱动任务继续执行。
49. 交互输入安全约束：
   - 密码类输入不得以明文持久化到聊天消息正文；
   - 过程轨迹可记录“已提交”状态与脱敏摘要，但不得记录明文密码；
   - 任务结束后应清理本轮临时交互敏感数据，避免跨任务泄漏。
50. 聊天任务有效性收敛、终态占位文案收敛与前后端展示一致性要求，统一以 `doc/01-or/【智能对话】智能体规划和回复的前后端处理.md` 为准。
52. 聊天任务运行态心跳时间需保证时区语义一致：
   - 运行态 `heartbeat_ts` 的写入与解析必须采用一致时区基准，避免因时区偏移导致看门狗误判超时；
   - 历史无时区时间格式需具备兼容解析策略，优先按“更接近当前时刻”的候选时区解释，避免任务实际运行中被误终止。
53. 聊天过程步骤时间展示口径与时区一致性要求，统一以 `doc/01-or/【智能对话】智能体规划和回复的前后端处理.md` 为准。
54. 智能体执行边界统一收敛到工具集层：
   - 所有智能体（含自主规划策略 `auto/react/cot/plan_execute/reflexion`）只能调用 `tools.*` 工具集接口，禁止在智能体代码中直接调用底层 `component.*`；
   - 自主规划执行链路中的模型调用、工具调用、组件清单查询等能力，需通过工具集统一代理后再进入组件层；
   - 若智能体尝试绕过工具集直连 `component.*`，需在运行时返回可读错误并停止本次调用。
55. `tools` 层需提供统一文件操作工具，支持文件读取、创建、覆盖写、追加写、文本替换等能力；写入相关操作需支持父目录自动处理（按需自动创建），并通过统一结构化结果返回执行状态。
56. `tools` 层文件操作工具必须复用 `component/handle/file_reader_component` 与 `component/handle/file_writer_component`，不得在工具层重复实现底层文件读写逻辑。
57. `tools` 层需提供路径管理工具，支持文件或文件夹的新建、重命名、移动能力；该工具明确禁止提供删除能力（文件删除、目录删除、递归删除均不允许），以降低高危误操作风险。
58. 非工具集模块（含 `core/*`、`agents/*`）禁止直接 `from tools.<toolset> import <ToolClass>` 硬编码导入具体工具类；工具调用需采用“工具集动态注册 + 运行时动态加载”方式：
   - 工具集存在性与启停状态优先通过 `tools.manager` 校验；
   - 运行时按工具集标识动态解析模块并调用；
   - 当工具集目录被删除、未注册或已停用时，业务层需返回可读错误，不得在 Django 启动阶段触发 `ModuleNotFoundError`。
59. 工具集列表页仅承担“工具集查询与启停管理”职责，不再提供“知识库整理”独立操作栏；相关介绍文案、执行按钮与结果展示区域均需移除，避免与工具集管理职责混用。
60. `tools` 层需提供 `create_tool` 脚手架工具，用于按传入内容创建新工具目录与关键文件（至少包含 `__init__.py`、`README.md`、主功能代码文件）；目录与文件写入必须复用现有 `tools/file_path_tool` 与 `tools/file_operator_tool`，不得在新工具中重复实现底层文件创建与写入逻辑；新工具创建路径必须固定收敛在当前仓库 `tools` 目录下（与 `create_tool` 自身目录同级），不允许通过外部入参将工具创建到其他目录。
61. 工具集需维护“工具源”字段并支持页面展示与筛选，来源枚举为：`原生`、`自生成`、`导入`；规则如下：
   - `create_tool` 创建的工具集默认标记为 `自生成`；
   - 通过导入流程落盘的工具集标记为 `导入`；
   - 未显式标记来源的存量工具集默认按 `原生` 处理；
   - 工具集列表页需新增“工具源”下拉筛选并展示每个工具集的来源标签。
62. 工具集列表页与智能体列表页的条目布局需保持紧凑：常用元信息应优先在同一行（允许自动换行）展示，减少低信息密度条目占用的垂直空间，保证单屏可见项数量提升。
61. 主聊天窗口需支持“@伙伴接管回复”：
   - 当用户在主聊天输入中 `@` 提及某个已启用伙伴时，本轮任务执行需按该伙伴配置生效（角色设定、记忆、模型、心法、工具与组件权限）；
   - 仅切换本轮执行主体，不迁移会话，消息继续留在主聊天窗口；
   - 聊天消息区“回复者名称”需与本轮实际执行主体一致：命中 `@伙伴` 时显示该伙伴名称，未命中时显示默认“助手”；
   - 当 `@` 目标不存在、未启用或无权限时，系统需回退主聊天默认助手语义，避免任务中断。
62. 聊天窗口需支持“单条消息删除”能力：
   - 主聊天窗口与伙伴聊天窗口均需提供消息删除入口；
   - 支持删除任意一条消息（用户消息与助手消息均可）；
   - 删除后仅移除目标消息本身，不触发整段会话回退；
   - 删除操作需按会话归属做权限校验，禁止跨用户删除。

## 非功能性要求
1. 目录命名需保持清晰、稳定、可读，避免同义重复目录并存。
2. 结构调整应优先兼容已存在代码与文档，避免大幅偏离既有实现。
3. 项目结构应具备可扩展性，允许在不破坏一级职责边界的前提下逐步细化子目录。
4. 文档内容聚焦结构与职责定义，不展开具体实现细节。
5. 组件化目录应保证命名和层级稳定，便于长期迭代中的组件替换与版本演进。

# 当前工作项
**需求**
