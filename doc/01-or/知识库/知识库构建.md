# 🧠 OpenClaw 本地全能助理系统 —— 增强版设计方案（v1.1）

> 在原有基础上，新增 **短期记忆（Working Memory）** 与 **长期价值存储（Insight Bank）**，实现从“资料库”到“会学习的智能体”的跃迁。

## 一、三层记忆模型

| 层级 | 名称 | 内容 | 生命周期 | 存储方式 |
|------|------|------|--------|--------|
| **L1** | **短期记忆（Working Memory）** | 当前会话上下文、用户意图、临时状态 | 单次会话 or 几小时 | 内存 + 临时文件（可选持久化） |
| **L2** | **结构化知识库（Structured Knowledge）** | 笔记、任务、项目、关系（原方案核心） | 长期、可编辑 | 本地文件 + SQLite（`index.db`） |
| **L3** | **长期价值信息（Insight Bank）** | 提炼的模式、规则、最佳实践、错误教训 | 永久、持续进化 | 专用目录 + 向量嵌入（可选） |

## 二、架构增强：新增 L1 与 L3

```
entities/

    │   └── sessions/           # 会话历史（按 session_id）

        └── index.db            # 专用索引（支持语义检索）
```

## 三、各层详细设计

### 🔹 L1：短期记忆（Working Memory）

#### 目标
- 记住当前对话的上下文（避免重复提问）
- 跟踪用户当前目标、未完成操作
- 支持多轮交互状态管理

#### 存储格式（`memory/sessions/{session_id}.json`）
```json
{
  "session_id": "sess_20260313_abc",
  "user": "you",
  "created_at": "2026-03-13T21:00:00",
  "context": {
    "current_task": "task:build_openclaw_assistant",
    "last_query": "如何实现文件监听？",
    "intent": "implement_file_watcher",
    "temp_variables": {
      "watched_dir": "./vaults/work/notes"
    }
  },
  "history": [
    {"role": "user", "content": "我想监控笔记变化"},
    {"role": "assistant", "content": "建议使用 watchdog 库..."}
  ]
}
```

#### 特性
- **自动过期**：会话超过 24 小时自动归档或删除
- **上下文注入**：每次生成回复前，将 `context` 注入 LLM prompt
- **可持久化**：重要会话可手动保存为笔记（→ L2）

---

### 🔸 L3：长期价值信息（Insight Bank）

#### 目标
- 从 L1（会话）和 L2（任务/笔记）中**自动提炼高价值信息**
- 存储**可复用的模式**，而非原始记录
- 支持未来**主动建议**（如“你上次遇到类似问题时用了方案 X”）

#### 内容类型

| 类型 | 示例 | 来源 |
|------|------|------|
| **错误模式** | “SQLite locked 错误 → 启用 WAL 模式” | 任务失败日志 + 用户反馈 |
| **高效流程** | “新建项目四步法：1.建 vault 2.写 config...” | 多次成功任务总结 |
| **决策规则** | “若任务涉及 API → 优先参考 FastAPI 最佳实践” | 知识图谱推理 |
| **个人偏好** | “你偏好使用 Whoosh 而非 Elasticsearch” | 历史选择统计 |

#### 存储格式（`insights/patterns/sqlite_locked.md`）
```markdown
---
id: pattern:sqlite_locked
title: "SQLite Database Locked 错误处理"
category: database
confidence: 0.95          # 置信度（基于出现频率）
source_sessions: ["sess_2025...", "sess_2026..."]
related_tasks: ["task:fix_db_lock"]
tags: [sqlite, concurrency, error]
---

## 触发条件
- 多线程写入 SQLite
- 未使用 WAL 模式

## 解决方案
1. 执行 `PRAGMA journal_mode=WAL;`
2. 缩短事务时间
3. 使用连接池（谨慎）

## 验证结果
> “启用 WAL 后，锁错误消失” —— 2026-02-10
```

#### 自动提炼机制（可选）
- **规则**：当同一错误在 3 个不同会话中出现 → 自动生成 `pattern`
- **AI 辅助**：用 LLM 总结会话：“请从以下对话中提取一条可复用的最佳实践”

---

## 四、三层协同工作流

### 场景：你再次遇到 SQLite 锁问题

1. **L1（短期记忆）**  
   - 当前会话记录：“又遇到 sqlite locked”
2. **L2（结构化知识）**  
   - 检索 `notes/` 和 `tasks/` → 找到过去的相关任务
3. **L3（长期价值）**  
   - 匹配到 `pattern:sqlite_locked`（高置信度）
   - 主动建议：“检测到你可能遇到 SQLite 锁问题，推荐以下方案...”
4. **反馈闭环**  
   - 若你采纳方案并成功 → 更新 `pattern` 的 `confidence += 0.05`
   - 若失败 → 创建新会话，未来可能生成更优 `pattern`
