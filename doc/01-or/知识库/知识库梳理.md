# 背景
当前工程已经具备会话收集、规则管理、组件管理等能力，但“知识资产”仍主要依赖人工整理，存在三类核心问题：
- 知识分散：笔记、任务、会话信息缺乏统一组织，复用成本高。
- 关系缺失：知识之间、知识与任务之间关联弱，难以在新问题中快速复用历史经验。
- 价值沉淀不足：历史会话中可复用模式无法稳定沉淀，系统长期学习能力弱。

结合现有两份分析材料，知识库能力需要从“静态存放”升级为“可分析、可整理、可持续优化”的本地能力体系，并与现有项目目录和工具规范对齐。

# 目标
1. 建立知识库原始需求边界，明确短期记忆、结构化知识、长期洞察三层能力定位。
2. 支持自动发现知识冗余、缺失关联和过期内容，降低知识库维护成本。
3. 支持基于现有知识与任务历史生成可执行优化建议，提升问题复用效率。
4. 支持输出周期性知识库健康报告，形成“发现问题-给出建议-人工确认-持续优化”的闭环。
5. 保持本地可控与可审计，确保知识处理过程可追溯、可人工介入。

# 要求
## 功能性要求
1. 系统需支持三层知识能力模型：
   - L1：短期记忆（会话上下文与临时状态）；
   - L2：结构化知识（笔记、任务、关系）；
   - L3：长期价值信息（模式、规则、经验洞察）。
2. 系统需提供知识库健康分析能力，至少输出：
   - 孤立知识项数量；
   - 标签分布与高频主题；
   - 疑似陈旧知识项列表。
3. 系统需提供关系补全建议能力，至少支持：
   - 基于关键词或文本重叠识别潜在关联；
   - 输出“来源项 -> 候选关联项 -> 置信说明”。
4. 系统需提供重复内容检测能力，支持识别高相似候选并给出保留建议。
5. 系统需支持根据分析结果生成可读的 Markdown 优化报告，包含：
   - 新增关系建议；
   - 重复内容建议；
   - 陈旧内容清单；
   - 关键洞察摘要。
6. 系统需支持手动触发执行，首期不强制引入后台常驻任务。
7. 系统输出需以本地文件为主，允许人工审核后再进入后续流程。

## 非功能性要求
1. 需优先复用现有工程目录与代码规范，避免重复建设平行能力。
2. 首期实现需保持轻量，不依赖外部云服务即可运行。
3. 输出结果应具备可解释性，建议内容需附带来源依据或命中条件。
4. 处理异常（目录不存在、文件编码异常、空知识库）时需返回可诊断错误信息。
5. 模块边界清晰，后续可平滑扩展到语义检索与自动提炼能力。
