## 问题描述
在能力检索中，`tool_function` 依赖固定 `_TOOL_FUNCTION_CATALOG` 维护函数清单。  
当新增工具目录或工具 README/API 发生变化时，若未同步修改该常量，前端工具查询与自主规划可见能力会出现不一致，导致“工具已存在但检索不到/未注册”。

## 关联文档
| 类型 | 文档路径 |
|:---|:---|
| 原始需求 | `codiiy/doc/01-or/复杂任务理解和处理的方案.md` |
| 开发设计 | `codiiy/doc/02-dr/01 会话原始需求收集系统设计.md` |
| 测试设计 | `codiiy/doc/03-tr/01 会话原始需求收集系统测试需求.md` |

## 根因分析
1. `tool_function` 目录来源绑定到静态常量，属于“人工同步目录”，不是“工具目录事实源”。  
2. 工具目录真实元数据已存在于 `tools/*/README.md` 与代码实现中，但检索层未消费该元数据。  
3. 结果是“工具目录状态”与“能力索引状态”分叉，形成耦合点与漏配风险。

## 涉及文件
- `codiiy/core/collector/orchestration/capability_search.py`
  - 移除静态 `_TOOL_FUNCTION_CATALOG` 依赖。
  - 新增工具函数动态发现：优先解析 `README.md` API 段落，代码反射补齐公共函数。
  - `tool_function` 统一由工具目录实时构建。
- `codiiy/test/test_core/test_collector/test_orchestration/test_orchestration.py`
  - 新增回归测试：验证 README API 自动发现与“无静态目录”构建。

## 设计约束更新
- 能力检索层禁止新增固定工具函数清单常量作为主数据源。
- 工具能力目录必须来自工具目录自描述（README/API）与代码反射，确保前端查询与自主规划共享同一事实源。
