# 质量保障与问题记录 (Quality Assurance, QA) 规范

**文件路径**: `doc/91-qa/AGENTS.md`
**目的**: 定义 Bug 反馈、问题分析与修复过程的文档化标准。
**状态**: 已采纳 (Adopted)

## 目录用途

本目录存放所有**用户反馈的问题/Bug 的分析和解决过程记录**。当用户报告问题或发现 Bug 时，相关的分析、根因定位、修复方案和涉及文件变更必须在本目录中形成文档，作为团队知识沉淀。

## 文档范围

### 允许放入
- Bug 分析与修复记录
- 问题排查过程文档
- 生产环境异常分析
- 回归问题记录

### 不应放入
- 原始需求（属于 01-or）
- 开发设计方案（属于 02-dr）
- 测试用例设计（属于 03-tr）
- 运维部署文档（属于 ops）

## 编辑规则

### 文档命名
- 格式：`【主题分类】问题简述.md`
- 主题分类应与功能模块对应，便于按模块检索。
- 示例：`【预标注】标签匹配问题.md`、`【点云标注】3D立方体框大小修改无法保存问题.md`

### 章节约束
每个 QA 文档**至少**包含以下章节：

| 序号 | 章节标题 | 目的 | 内容要求 |
| :--- | :--- | :--- | :--- |
| 1 | # 问题描述 | 清晰记录问题现象 | 描述问题现象、复现条件、影响范围 |
| 2 | # 关联文档 | **建立双向追溯链** | 用表格列出该问题关联的原始需求(01-or)、开发设计(02-dr)、测试设计(03-tr)文档路径，实现从问题到需求/设计的完整追溯 |
| 3 | # 根因分析 | 定位问题根本原因 | 说明根本原因，包含必要的代码引用、日志、调用链分析 |
| 4 | # 涉及文件 | 记录修改范围 | 列出修改的文件及修改内容摘要（建议用表格） |
| 5 | # 设计约束更新（可选） | 同步设计变更 | 若修复涉及设计变更，说明对应设计文档的更新内容 |

### 变更流程
- 收到用户 Bug 反馈后，**必须**先在本目录创建文档，再进行代码修复。
- 创建文档时**必须**填写 `# 关联文档`，定位到对应的原始需求和设计文档；若暂时无法确定，标注"待定位"并在分析完成后回填。
- 修复完成后，回填根因分析和涉及文件等内容，并**最终确认**关联文档章节完整。
- 若修复涉及设计变更，需同步更新 02-dr 中的设计文档，并在 QA 文档中引用。

## 协作约定

| 相邻目录 | 关系 |
| :--- | :--- |
| 01-or | 若 Bug 暴露出需求遗漏，应在 01-or 中补充需求 |
| 02-dr | Bug 修复涉及设计变更时，需同步更新设计文档 |
| 03-tr | Bug 修复后应在 03-tr 中补充回归测试用例 |
