# 自主规划假死和死循环

## 过程可观测性缺口（2026-03-15）
- 现状问题：前端“处理中间步骤”只展示 `step_agent_mindforge` 外层事件；心法步骤内部的大模型调用与工具调用被隐藏，导致页面长时间停在单一步骤、用户误判“卡死”。
- 设计约束：自主规划链路必须支持“步骤内细粒度事件流”，至少覆盖每次大模型调用与每次工具调用的 `start/end/result` 三段事件，并实时透传到 `process_trace.events`。
- 透传链路：`Coordinator(step)` -> `mindforgeRunner` -> `strategy`/`ReActEngine`，事件模型统一为 `kind/title/status/input/output/error/ts`，禁止仅在步骤完成后一次性回填。
- 前端展示约束：即使仍处于同一外层步骤，也必须按内部事件逐条刷新轨迹列表，确保用户可见“正在调用哪个模型/哪个工具、已完成多少次、最近一次结果是什么”。
- 兼容约束：保留现有外层步骤事件，不替换；新增内部事件作为增量补充，避免影响旧任务回放与历史渲染逻辑。

# 多轮对话之间的上下文和记忆问题
问题本质
“同一个聊天会话” ≠ “同一个 Python 进程状态”
如果你的系统每次请求都是无状态 HTTP 调用（如 FastAPI/Flask），那么：
每次用户发消息，后端都启动一个新进程
上一轮的 plan、session_id、file backups 等全部丢失

# auto自主规划重规划的问题
当前重规划只能基于工具调用或者运行异常，如果调用运行成功，则不会重规划。
