# 测试需求与设计 (Test Requirements, TR) 规范

**文件路径**: `doc/03-tr/AGENTS.md`
**目的**: 定义测试需求与测试设计文档的管理标准，确保测试与开发设计同步推进。
**状态**: 已采纳 (Adopted)

## 目录用途

本目录存放所有功能的**测试需求分析和测试用例设计**文档，是开发设计（02-dr）的对等产出物。每个功能模块在进行开发设计的同时，必须在本目录中产出对应的测试设计。

## 文档范围

### 允许放入
- 测试需求分析文档
- 测试用例设计文档（含手动测试用例和自动化测试用例说明）
- 测试策略说明（如覆盖范围、优先级、框架选型等）

### 不应放入
- 原始需求（属于 01-or）
- 开发设计方案（属于 02-dr）
- Bug 分析与修复记录（属于 91-qa）
- 测试执行报告、测试数据等运行时产物

## 编辑规则

### 文档命名
- 与 `02-dr` 中对应的开发设计文档保持一致的模块分类，便于对照查找。
- 格式示例：`doc/03-tr/06 预标注/预标注业务流程测试设计.md`

### 目录组织
- 按功能模块建立子目录，子目录编号和名称与 `02-dr` 保持对应。
- 例如 `02-dr/06 预标注/` 对应 `03-tr/06 预标注/`。

### 章节约束
测试设计文档至少包含：
1. **关联文档**（强制） — 用表格列出对应的原始需求(01-or)和开发设计(02-dr)文档路径，建立需求→设计→测试的完整追溯链。
2. **测试范围** — 覆盖哪些功能点、接口、页面。
3. **测试用例** — 表格或列表形式，每条包含：前置条件、操作步骤、预期结果。
4. **自动化测试标识** — 标注哪些用例需自动化，使用的框架（WEB 类统一使用 Playwright）。

### 变更流程
- 开发设计文档变更时，必须同步检查并更新对应测试设计文档。
- 新增功能模块时，先建立测试设计文档框架，再逐步填充用例。

## 协作约定

| 相邻目录 | 关系 |
| :--- | :--- |
| 01-or | 测试设计的输入来源：从原始需求中提取测试点 |
| 02-dr | 对等关系：开发设计与测试设计同步产出，功能模块一一对应 |
| 91-qa | Bug 修复后若涉及新增回归用例，应在本目录补充 |

## 自动化测试框架约定

| 测试类型 | 框架 | 代码位置 |
| :--- | :--- | :--- |
| WEB 系统级测试（E2E） | Playwright | `test/e2e/` |
| 后端单元/集成测试 | pytest | `test/` （按 rules-test.md 映射规则） |
