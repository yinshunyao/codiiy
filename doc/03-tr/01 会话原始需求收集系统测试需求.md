## 1. 测试范围
1. 登录能力
2. 密码修改能力
3. 会话创建与列表展示
4. 多轮消息发送与展示
5. 单条消息删除能力（主聊天/伙伴聊天）
6. SQLite 持久化能力
7. 默认管理员初始化能力
8. 项目管理页展示约束（顶部不展示提示信息）
9. 工具集列表与智能体列表紧凑布局展示
10. 菜单命名方案切换（标准中文/武侠风）
11. 组件子菜单命名映射（沟通/观察/决策/操作）
12. `auto` 预热判定与策略衔接（直接完成/继续执行）
13. `auto` 对 `react/plan_execute` 的工具上下文与 API 调用提示
14. `mindforge_runner` 动态预算（复杂任务与简单任务的 `max_steps/max_tokens` 梯度）
15. `coordinator` 工具回退重试链路（主目标 + fallback，尝试上限可配置）
16. `coordinator` 不可恢复错误停止重试（授权/白名单/参数硬失败）
17. 工具步骤结果可观测字段（`selected_function_path/attempt_count/fallback_used/attempt_trace`）
18. 工具函数索引动态发现（README API + 代码反射）一致性
19. ReAct 工具注册白名单兜底（检索未命中时仍可执行）
20. ReAct 工具返回结构兼容（`data.result`/直接 `data`）

## 2. 测试项
1. 首次迁移后，`admin / 123456` 可登录。
2. 未登录访问 `/sessions/` 会跳转到登录页。
3. 登录后可访问会话列表和新建页。
4. 新建首条消息后自动创建会话，列表可见该记录。
5. 在会话详情连续发送多条消息，页面按时间顺序展示多轮对话。
6. 每次发送用户消息后，系统自动回复固定文本 `收到`。
7. 修改密码成功后，旧密码失效，新密码生效。
8. 重启服务后，会话与消息数据仍存在（SQLite 持久化）。
9. 执行“设为默认项目”等会写入消息队列的操作后，`/projects/` 页面顶部不展示提示信息文本列表。
10. 访问工具集列表页时，条目模板应输出紧凑布局结构（标题操作同行、元信息标签行）。
11. 访问智能体列表页时，条目模板应输出紧凑布局结构（目录/更新时间等元信息合并展示）。
12. 在主聊天中点击任意消息“删除”后，目标消息应被移除，其余消息不受影响。
13. 在伙伴聊天中点击任意消息“删除”后，目标消息应被移除，并返回当前伙伴聊天页。
14. 非消息归属用户不可删除该消息（返回 404 或权限失败），避免跨用户越权删除。
15. 会话详情中用户消息的“回退/删除”（启用总结时含“摘”）按钮应位于同一操作行，避免分两行展示。
16. 个人设置页应展示“菜单命名方案”选择器，默认值为“武侠风”。
17. 提交菜单命名方案后，侧栏一级菜单文案应即时切换（如“聊天”->“江湖”）。
18. 武侠风下，工具集入口应显示为“兵刃”，组件一级菜单显示为“机枢”。
19. 武侠风下，组件二级菜单应显示为“传音/望气/运筹/出招”。
20. 菜单映射配置文件缺失或异常时，系统应回退为标准中文文案，不影响页面渲染。
21. 武侠风下系统品牌名称应显示为“开智枢”，标准中文下应显示为“codiiy平台”。
22. “侠谱/伙伴管理”应位于“武库/智能体”分组下且在“兵刃/工具集”之前。
23. 侧栏不展示伙伴创建“+”按钮；“侠谱”页面内应保留创建入口。
24. `auto` 预热返回 `is_completed=true` 且附带答案时，应直接结束并返回该答案，不再进入后续策略执行。
25. `auto` 预热返回 `next_strategy=cot` 时，应按 CoT 链路继续，不应误触发 `react/plan_execute`。
26. `auto` 预热建议与执行型任务约束冲突时（如建议 `cot` 但任务是执行型），应触发护栏纠偏并记录原因。
27. `react` 阶段提示词应包含可用工具 API 调用方式（`{"tool":"...","kwargs":{...}}`），确保工具调用格式可执行。
28. `plan_execute` 规划提示词应包含“多命令按顺序拆分执行”的约束，避免把多条命令塞在单步。
29. 复杂查询场景下，`mindforge_runner` 计算得到的 `max_steps` 与 `max_tokens` 应显著高于简单查询，且满足上限约束。
30. `tool` 步骤执行时，应按“主目标 -> fallback_targets”顺序尝试，成功后返回最终命中函数路径。
31. 当工具错误命中不可恢复类别（未授权/白名单拒绝/参数结构硬失败）时，应立即停止后续 fallback 尝试。
32. 当 `tool_max_attempts` 显式配置为 N 时，工具尝试次数不得超过 N 且不超过候选目标总数。
33. 工具步骤失败与成功两类结果都应包含 `attempt_count`、`fallback_used` 与完整 `attempt_trace`。
34. 总结阶段当 `completion_signal.need_replan=true` 且 `has_alternative=false` 时，仍必须触发一次重规划执行。
35. `capability_search` 应可从工具 `README.md` 自动识别 API 函数（如 `macos_terminal_tool.run_command`）。
36. `capability_search` 构建 `tool_function` 条目时，不依赖固定静态函数清单，新增工具后可自动进入索引。
37. `mindforge_runner` 在 `search_tool_functions` 无命中时，仍应从白名单工具集补齐函数注册，避免 `工具未注册`。
38. `capability_search` 代码反射仅提取工具主类公共方法，内部辅助类方法不得进入 `tool_function` 目录。
39. ReAct 执行器在工具返回 `{"success":true,"data":...}`（无 `result` 字段）时，`observation_preview` 不得为 `null`。
