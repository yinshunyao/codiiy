# 知识库整理构建模块设计

## 1. 设计目标
1. 在现有工程内提供一个本地可运行的知识库整理工具，支持健康分析、关系建议、重复检测、报告生成。
2. 以文件系统为首期数据源，兼容后续接入数据库索引与语义检索。
3. 保持模块低耦合，作为 `tools` 独立能力供后续系统流程复用。

## 2. 需求分析
对应 `doc/01-or/知识库/知识库梳理.md` 的核心需求：
- 功能性：三层记忆能力边界、健康分析、关系补全、重复检测、优化报告。
- 非功能性：本地可控、轻量、可解释、异常可诊断、可扩展。

本期范围（MVP）：
- 实现 L2（结构化知识文件）的整理与分析；
- 为 L1/L3 预留数据目录与字段接口，但不引入复杂调度与外部模型依赖。

## 3. 核心概念与术语
- **知识项（Knowledge Item）**：单个 Markdown/TXT 文档，包含路径、标题、文本、标签。
- **孤立知识项**：未与其他知识项形成任何建议关联的项。
- **重复候选**：文本相似度超过阈值（默认 0.9）的知识项对。
- **关联建议**：由关键词重叠与文本相似度综合得到的“可能有关联”结果。
- **健康报告**：汇总分析结果的 Markdown 文档。

## 4. 设计方案
### 4.1 模块结构
新增目录：
- `tools/knowledge_curation_tool/`
  - `knowledge_curation_tool.py`：核心实现；
  - `__init__.py`：导出入口；
  - `README.md`：使用说明与边界。

测试目录（镜像规则）：
- `test/test_tools/test_knowledge_curation_tool/test_knowledge_curation_tool.py`

### 4.2 核心流程
1. 扫描知识目录，收集 `.md/.txt` 文件。
2. 解析文档元信息：
   - 标题：优先首个 `#` 标题，否则使用文件名；
   - 标签：支持文内 `#tag` 提取；
   - 文本：保留可读文本，过滤空白。
3. 执行分析：
   - 健康分析（孤立项、标签统计、疑似陈旧）；
   - 关系建议（关键词交集 + 文本相似度阈值）；
   - 重复检测（高相似内容对）。
4. 生成优化报告（Markdown），输出到指定目录。

### 4.3 数据结构设计
`KnowledgeItem`（dataclass）：
- `item_id: str`
- `path: str`
- `title: str`
- `content: str`
- `tokens: set[str]`
- `tags: list[str]`
- `modified_at: float`

返回结构统一：
- 成功：`{"success": True, "data": {...}}`
- 失败：`{"success": False, "error": "..."}`

### 4.4 接口设计（Python）
`KnowledgeCurationTool` 提供方法：
- `scan_items(knowledge_dir)`
- `analyze_health(knowledge_dir, stale_days=180)`
- `suggest_links(knowledge_dir, min_shared_tokens=2, min_similarity=0.35, top_k=3)`
- `detect_duplicates(knowledge_dir, similarity_threshold=0.9)`
- `generate_report(knowledge_dir, output_dir, report_name=None)`

### 4.5 技术选型与理由
- 仅使用 Python 标准库（`os`、`re`、`dataclasses`、`difflib`、`collections`、`datetime`）：
  - 优点：零额外依赖、部署简单；
  - 缺点：语义能力有限，后续可演进为向量检索。

## 5. 异常处理与边界情况
1. 目录不存在：返回错误，不抛出未处理异常。
2. 空目录：返回空结果与可读提示。
3. 文件编码异常：使用 `errors="replace"` 读取，避免中断全量扫描。
4. 超大文件：首期不做硬限制，后续可加入大小阈值与截断策略。

## 6. 安全设计
1. 仅在指定目录内读取文件，不执行动态代码。
2. 报告文件名采用受控命名，不拼接不可信路径片段。
3. 不上传外部服务，不出网处理内容，满足本地可控要求。

## 7. 部署与配置
1. 无新增系统依赖。
2. 工具作为本地 Python 模块直接引入运行。

## 8. 监控与日志
1. 本期以函数返回结果为主，不引入统一日志管道。
2. 失败场景在 `error` 返回中包含可诊断信息。

## 9. 测试策略
1. 使用 `unittest` 构建最小回归：
   - 扫描能力；
   - 关系建议命中；
   - 重复检测命中；
   - 报告生成成功。
2. 采用临时目录构造样例数据，避免依赖真实仓库内容。

## 10. 回滚方案
1. 新增模块为增量能力，不改动现有业务流程。
2. 如需回滚，仅移除 `tools/knowledge_curation_tool` 与对应测试。

## 11. 页面接入设计（core/collector）
1. 接入页面：`core/templates/collector/toolset_list.html`（工具集页面）。
2. 交互方式：
   - 在工具集页面增加“知识库整理”功能卡片；
   - 提供“一键执行知识库整理”按钮（POST 提交）；
   - 执行完成后在同页展示报告路径、核心统计与报告文本预览。
3. 后端入口：
   - 复用 `toolset_list` 视图处理 GET/POST；
   - GET 负责列表与筛选；
   - POST 识别 `action=run_knowledge_curation` 后调用 `KnowledgeCurationTool.generate_report`。
4. 默认路径策略（按当前项目）：
   - `knowledge_dir`: `<project.path>/doc/01-or`
   - `output_dir`: `<project.path>/data/temp/knowledge_reports`
5. 错误处理：
   - 工具不可导入、目录不存在、执行失败时，页面展示可读错误信息；
   - 不中断工具集列表渲染。
