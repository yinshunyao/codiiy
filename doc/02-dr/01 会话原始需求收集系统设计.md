# 会话原始需求收集系统设计

## 1. 设计目标
1. 提供一个基于 Web 的原始需求入口与原始需求列表入口。
2. 提供基础账号能力：登录、修改密码。
3. 数据持久化：当前版本使用 SQLite。
4. 最大化复用 Django 内置能力，减少重复开发。

## 2. 技术方案
1. 后端框架：Django
2. 认证能力：`django.contrib.auth` + `django.contrib.sessions`
3. 页面渲染：Django Templates
4. 数据库：SQLite（Django ORM 管理）
5. 管理后台：Django Admin（用于辅助管理会话数据）

## 3. 功能设计
### 3.1 认证
1. 登录页：使用 Django `LoginView`。
2. 退出登录：使用 Django `LogoutView`。
3. 修改密码：使用 Django `PasswordChangeView` + `PasswordChangeDoneView`。
4. 默认管理员：通过数据迁移初始化 `admin / 123456`（仅当用户不存在时创建）。

### 3.2 会话需求收集
1. 会话实体：`RequirementSession`
- `title` 标题
- `created_by` 创建人
- `created_at` 创建时间
- `updated_at` 更新时间
2. 消息实体：`RequirementMessage`
- `session` 所属会话
- `role` 角色（`user` / `assistant`）
- `content` 消息内容
- `created_at` 创建时间
3. 页面能力：
- 会话列表页：展示当前用户创建的会话
- 会话详情页：展示单条会话的多轮消息
- 新建会话：通过首条用户消息自动创建会话
- 原始需求列表页：展示当前项目 `doc/01-or` 下的文件列表，支持从会话页顶部入口点击跳转
- 路径容错：当当前项目路径无权限或不可访问时，文件列表页回退到系统默认目录 `doc/01-or` 并提示用户，避免页面 500
3. 访问控制：
- 列表/详情/新建必须登录后访问

### 3.3 UI 与交互设计（对话产品风格）
1. 整体布局采用“左侧导航 + 中间对话主区 + 右侧历史会话”三栏结构：
- 左侧：一级/二级菜单导航（聊天、组件、设置等）
- 中间：当前会话内容与输入区
- 右侧：历史会话列表（按最近更新时间倒序）
2. 视觉风格：
- 深色主题、圆角卡片、柔和阴影、低对比边框
- 输入区固定在主内容底部，符合主流大模型对话页输入习惯
 - 消息气泡高度随内容自适应，不设置固定高度
 - 对话内容区设置居中聊天轨道，减少左右气泡视觉距离
3. 交互规则：
- 左侧“聊天”一级菜单下仅保留一个二级入口“聊天”；
- 点击二级“聊天”默认进入当前对话；若当前无激活会话则进入最近一次会话；
- 移除侧栏独立“新对话”按钮，在二级“聊天”项内提供 `+` 新建入口；
- 历史会话列表统一放到右侧栏，点击后切换到对应会话详情；
- 一级菜单支持点击展开/收缩，默认展开当前常用分组；
- 左侧导航定义为系统级公共菜单：在“设置”分组点击“项目管理”“原始需求列表”等二级菜单后，左侧菜单结构与样式保持不变，不允许切换为页面私有侧栏；
- 当前页面所属菜单项需高亮显示；当处于设置类页面时，“设置”分组默认展开并保持可见；
- 侧栏底部用户信息区按钮文案保持简短（如“在线”“退出”），避免冗长标签影响视觉密度；
- 新建会话后跳转到对应会话详情页
- 在会话详情页持续发送消息，形成多轮对话
- 未选择会话时展示引导文案和快捷操作
 - 用户发送消息并完成页面刷新后，右侧对话区自动滚动到底部，展示最新用户消息和助手回复
 - 左侧会话列表与右侧对话区独立滚动，互不影响
- 顶部标识“原始需求列表”支持点击跳转到文件列表页
- 文件列表页默认展示当前项目原始需求目录下的文件名、创建日期、更新时间、文件大小（自动换算为 KB/MB 等单位）
- 点击 `.md` 文件名在当前页面加载预览区，展示该文件内容
4. 消息展示约束：
- 消息气泡高度必须随内容自适应，不允许固定高度。
- 气泡本身不保留模板缩进空白，避免因空白符导致异常拉高。
- 消息正文渲染需要对模板缩进空白做归一化处理：保留用户真实换行，不保留模板层面的前导/尾随空白。
- 对话轨道宽度需收敛到可读区间（建议 `720~860px`），在大屏下保持居中，避免消息左右跨度过大。
- 单条气泡最大宽度建议不超过轨道的 `78%`，并设置合理最小宽度，保证短消息不显得过窄、长消息不贴边。
- 对话区需设置统一上下留白与行间距，消息间垂直节奏保持一致，减少“松散/拥挤”观感。
 - 语音消息展示规则：若消息包含音频附件，先展示语音播放器；若存在识别文本，默认折叠不展开，用户点击“展开文字/收起文字”按钮后切换可见性。
5. 大模型选择器：
- 侧栏使用单一按钮展示当前大模型名称，按钮样式与“当前项目-管理”按钮保持一致。
- 点击按钮展开一级菜单（厂商列表）。
- 鼠标悬停一级菜单项时，在右侧展示二级菜单（该厂商下模型列表）。
- 点击具体模型后即时保存到当前项目，并更新按钮文本为新模型名称。
- 模型按钮与“当前项目”放在同一栏位头部，不再使用独立栏位。
- 左侧栏移除“当前项目”模块，顶部标题栏改为直接展示当前项目名称（位于页面标题右侧），并保留项目管理入口。
- 为避免悬停丢失，二级菜单与一级菜单之间不保留悬停断点，并提高菜单层级，确保可点击。
- 内置阿里模型清单并写入数据库（不存在则补齐），前端直接展示该厂商下全部内置模型供选择。
- 一级菜单字体与现有侧栏风格保持一致（字号、颜色、间距对齐 `chip/meta` 风格），避免视觉突兀。
- 交互实现采用稳定联动：一级厂商列表与右侧二级模型列表同容器双列展示，悬停厂商时右侧即时刷新模型列表。
- 下拉菜单需要以浮层方式显示，层级高于右侧主内容区域，避免二级模型列被页面主体遮挡。
- 布局上保留左侧会话侧栏，模型切换入口放置到主内容顶部左侧区域；仅调整布局，不改变现有视觉主题。
6. 语音输入交互：
- 会话输入区增加麦克风按钮；点击后开始录音，再次点击结束录音。
- 录音文件以音频附件方式随消息一起发送（复用现有附件上传通道）。
- 语音识别采用浏览器内置语音识别能力，将识别文本回填到输入框并随消息提交到后端保存。
- 若浏览器不支持语音识别或识别失败，仍允许用户仅发送音频附件。
7. 输入区布局规范（参考主流对话产品）：
- 输入区采用圆角胶囊风格，单行主输入交互；整体保持低对比边框与浅背景。
- 左侧放置“+”附件入口，点击后触发附件选择。
- 中间为输入区域，占据主要宽度，提示文案简洁（如“有问题，尽管问”）。
- 右侧放置两个操作入口：麦克风按钮（语音识别）与圆形语音按钮（录音开始/停止）。
- 圆形语音按钮在录音中需有明显状态反馈（颜色或文案变化），避免误操作。

### 3.4 AI 回复策略（当前版本）
1. 会话采用两段式处理：
- 第一段（`collecting`）：目标是“搞清楚用户需求描述”，可持续追问澄清；当用户确认描述完成且模型判断信息完整后，进入第二段。
- 第二段（`organizing`）：目标是“整理原始需求文档”，基于会话历史和规则文档生成结构化原始需求内容。
2. 第一段回复策略：
- 若需求信息完整且无疑问，回复：`已收到，你描述完了么？`
- 若信息不完整，提出 2-3 个有针对性的澄清问题。
3. 第二段提示词策略：
- 系统提示词（`system`）中注入“角色约束、输出格式约束、规则文档内容”。
- 用户提示词（`user`）仅提供会话历史与“开始整理”的显式指令，避免规则信息散落在用户内容中。
4. 降级策略：
- 大模型不可用时，第一段保持确认式回复；第二段返回错误信息，不覆盖已有文档内容。
5. 第一段完成判定策略：
- 不仅识别固定短语 `已收到，你描述完了么？`，还应识别“需求已完成/已确认完成/可进入整理”等等价语义。
- 当用户明确表达“描述完了/说完了/完成了”等确认语义，且模型回复存在“完成语义”时，应进入第二段整理流程。
- 若模型回复中同时包含补充问题，不阻塞第二段触发，补充问题可在第二段文档确认环节继续处理。
6. 第一段输出信号协议：
- 第一段系统提示词强制模型返回两行结构化内容：`SIGNAL: ...` 与 `REPLY: ...`。
- `SIGNAL` 仅允许两种值：`READY_FOR_ORGANIZE`（可进入第二段）或 `NEED_CLARIFICATION`（继续澄清）。
- 后端优先读取 `SIGNAL` 进行阶段切换判断，语义关键词匹配作为兼容兜底。
7. 模型能力兼容策略：
- 第一段需求澄清与第二段需求整理当前均按文本对话调用。
- 当项目选择的模型为视觉模型（如 `qwen-vl-*`）且本次请求未构造多模态输入时，后端自动回退到文本模型（默认 `qwen-plus` 或系统配置模型）执行。
- 回退逻辑仅影响模型调用，不改变项目已选模型配置；并在日志中记录回退信息，便于排查。
8. 多模态消息构造规则（图片/音视频附件）：
- 会话消息在数据库中继续保存 `content` 与 `attachment`；构造模型请求时按“文本 + 附件”组装。
- 当消息包含可识别多模态附件（如 `image/*`、`video/*`、`audio/*`）且本次调用目标模型为视觉模型（`qwen-vl-*`）时，后端将该消息构造成多模态 `content` 列表：
  - 文本块：`{"text": "..."}`
  - 附件块需按模态字段构造（不使用通用 `file`）：
    - 图片：`{"image": "data:<mime_type>;base64,<encoded_content>"}`
    - 视频：`{"video": "data:<mime_type>;base64,<encoded_content>"}`
    - 音频：`{"audio": "data:<mime_type>;base64,<encoded_content>"}`
- 当消息仅有附件无文本时，仍允许发送附件块；当仅有文本无附件时，按纯文本消息发送。
- 若附件类型不属于视觉模型支持的模态（如 `pdf/doc/docx`），该附件不构造成视觉模型附件块，仅作为文本提示参与上下文。
9. 多模态与模型路由联动策略：
- 路由判断基于“所选模型 + 本次请求是否存在**可识别多模态附件块**”：
  - 视觉模型 + 存在可识别多模态附件块：保持视觉模型，不回退。
  - 视觉模型 + 不存在可识别多模态附件块：回退到文本模型（`qwen-plus` 或系统配置模型）。
  - 文本模型：始终按文本调用（附件不触发视觉路由）。
- 第一阶段（需求澄清）与第二阶段（需求整理）统一使用上述联动策略，保证日志与实际调用一致。
10. 模型响应解析兼容规则：
- 若响应同时包含 `text` 与 `choices[*].message.content`，且 `text` 为空（`null` 或空串），应优先使用 `choices` 中的消息内容。
- 当 `content` 为数组时，按顺序提取其中的 `text` 字段并拼接；当 `content` 为字符串时直接使用。
- 解析层必须保证返回字符串类型，避免 `None` 进入后续 `strip()` 引发异常。

### 3.5 规则读取与优先级设计
1. 新增通用工具 `tools/rule_reader`，供会话整理等模块复用。
2. 输入可以是目录或文件路径：
- 若输入为文件，从其所在目录开始向上逐级查找 `rules.md`。
- 若输入为目录，从该目录开始向上逐级查找 `rules.md`。
3. 层级扫描范围：
- 默认向上扫描到文件系统根目录；
- 支持传入 `stop_at`（建议传项目根目录）限制扫描上边界，避免跨项目读取。
4. 优先级规则：
- 距离目标路径最近的 `rules.md` 优先级最高；
- 返回结果按“高优先级 -> 低优先级”排序（即“近 -> 远”）。
5. 输出结构：
- `rules`：规则列表（每项包含路径、层级距离、内容）；
- `merged_rules`：可直接注入提示词的拼接文本（按优先级顺序分段）；
- `resolution_order`：优先级说明文本，明确“就近优先”。
6. 在需求整理阶段，优先使用 `merged_rules` 注入 `system` 提示词，确保模型按项目规则输出原始需求文档。

### 3.6 本地屏幕理解组件（component/observe）
1. 目标：
- 在 `component/observe` 下提供一个轻量、本地可运行的“截图 + 视觉理解”组件；
- 不依赖云端服务，依赖本机截图库与本地 vLLM 服务闭环完成推理；
- 适配消费级显卡（RTX 3060/4060）场景，优先小参数视觉模型（如 `Qwen2-VL-0.5B-Instruct`）。
2. 模块划分：
- `component/observe/screen_capture_component/api.py`：屏幕截图能力，默认使用 `mss`，支持区域截图与全屏截图；
- `component/observe/screen_understand_component/qwen2vl_vllm.py`：vLLM 推理客户端，调用本地 OpenAI 兼容接口（`/v1/chat/completions`）；
- `component/observe/screen_understand_component/api.py`：端到端编排（截图 -> 图片编码 -> 视觉理解）；
- `component/observe/__init__.py`：对外函数式 API 唯一导出入口；
- `component/observe/screen_capture_component/README.md`：截图组件说明；
- `component/observe/screen_understand_component/README.md`：理解组件说明。
3. 推理链路：
- 输入可为“截图路径”或“图像对象（PIL.Image）”；
- 若未提供图像输入，可直接执行实时截图；
- 将图像编码为 `data:image/png;base64,...` 后发送给本地 vLLM；
- 返回自然语言描述，或按 `json_mode=True` 返回结构化 JSON。
4. 本地 vLLM 部署约束：
- 使用 vLLM 启动本地模型服务，推荐命令：
  - `python -m vllm.entrypoints.openai.api_server --model Qwen/Qwen2-VL-0.5B-Instruct --served-model-name qwen2-vl-0.5b --max-model-len 4096 --dtype auto --gpu-memory-utilization 0.85`
- 默认访问地址：`http://127.0.0.1:8000/v1/chat/completions`；
- 组件通过可配置参数支持切换 `base_url`、`model`、`timeout`。
5. 性能与资源策略（轻量化）：
- 截图优先使用 `mss`（低开销、速度快），默认输出 PNG；
- 推理参数默认偏保守：`temperature=0.2`、`max_tokens=256`；
- 支持限制截图区域，减少图像尺寸与显存占用；
- API 采用懒初始化，不在 import 时加载重依赖。
6. 异常与边界处理：
- 本地 vLLM 不可用时抛出可读错误（连接失败、超时、非 200 响应）；
- 截图库缺失时提示安装依赖；
- 图片输入为空、路径不存在、JSON 解析失败时返回明确异常信息；
- 对模型返回进行兜底：优先取文本，`json_mode=True` 时先尝试 JSON 解析，失败则返回原始文本。
7. Demo 运行约定：
- `component/observe` 外层目录不再提供聚合 demo；
- 各组件在自身目录独立提供 `demo.py`（如 `screen_capture_component/demo.py`、`screen_understand_component/demo.py`）；
- 默认通过文件内变量配置运行参数，不依赖命令行参数；
- 组件 demo 必须使用绝对导入，确保可直接执行；
- Demo 仅用于开发验证，不参与 Django 主流程。
8. tools 工具封装约定：
- `tools/control_call_tool` 作为面向 `component` 的通用工具封装；
- 工具在调用截图/理解能力前，必须先读取并检查自身 `requirements.txt` 依赖；
- 若发现依赖缺失，工具先执行安装，再继续调用 `component` 层功能；
- 支持 `component` 整体信息查询（读取 `component/**/README.json`）；
- 支持特定函数路径调用，统一入口命名为 `control_call`；
- 工具返回统一字典结构：`success`、`data`、`error`。
9. 字符串路径调用约定：
- 新增 `component` 统一调用入口，支持通过字符串函数路径调用，例如：`component.observe.understand_current_screen`；
- 调用器负责完成“路径解析 -> 函数加载 -> 参数透传 -> 异常抛出”；
- 为降低误调用风险，函数路径必须以 `component.` 前缀开头。
10. component 目录自述约定（JSON）：
- `component` 及其每个子目录都必须包含一个 JSON 自述文件（`README.json`）；
- 自述文件需明确列出“函数路径 + 功能描述 + 入参/出参说明”；
- 当新增/调整函数时，同步更新对应目录 `README.json`。
- 依赖包导入名映射（如 `pillow -> PIL`）统一配置在 `component/README.json` 的工具规则字段中；
- tool 侧仅负责读取该规则并执行逻辑，不再新增独立映射配置。
12. component 调用规范去模糊：
- 不保留模块级兼容转发文件（如 `screen_capture.py` / `screen_understand.py`）；
- 对外调用仅使用 `README.json` 与 `component_index.json` 登记的规范函数路径；
- 升级时通过文档与索引显式发布变更，不通过隐式别名兜底。
11. Django 首页组件菜单与函数管理：
- 在首页左侧栏增加树形菜单：一级菜单 `组件（component）`，二级菜单按顺序展示 `沟通（communicate）`、`观察（observe）`、`决策（decide）`、`操作（handle）`；
- `component` 功能页左侧菜单不再单独实现，必须复用与会话页一致的侧栏菜单模板（单一引用），保证菜单结构、文案、交互（展开/收缩）一致；
- 点击二级菜单进入对应功能页，列表展示该模块函数信息（读取 `component/<module>/README.json`）；
- 功能页支持函数信息模糊搜索（函数路径、描述、输入/输出文本匹配）；
- 功能页支持模块目录下载：将 `component/<module>` 目录打包为 zip 下载；
- 功能页支持模块目录上传：上传 zip 后在后端解压覆盖到对应目录，并基于最新 `README.json` 刷新函数信息；
- 上传解压需进行路径安全校验，禁止 zip slip（目录穿越）写入目标目录外路径。

## 5. 数据库设计
### 5.1 表：collector_requirementsession
1. `id`：主键，自增
2. `title`：字符型，最大 200
3. `content`：文本型（兼容保留，不作为多轮消息主存储）
4. `created_by_id`：外键，关联 `auth_user.id`
5. `created_at`：创建时间，自动写入
6. `updated_at`：更新时间，自动更新

### 5.2 表：collector_requirementmessage
1. `id`：主键，自增
2. `session_id`：外键，关联 `collector_requirementsession.id`
3. `role`：字符型，`user` 或 `assistant`
4. `content`：文本型
   - 语音消息场景下，保存语音识别后的文本（允许为空，表示仅发送语音附件）
5. `created_at`：创建时间，自动写入

### 5.3 初始化策略
1. 通过迁移脚本创建默认管理员账号：
- 用户名：admin
- 密码：123456
- 权限：superuser + staff
2. 若系统中已存在用户，则不重复创建。

## 6. 安全与约束
1. 登录后访问控制使用 Django 装饰器/Mixin。
2. 密码修改后继续保持登录状态（默认行为）。
3. 开发环境允许 `DEBUG=True`，生产需关闭并配置安全项。
