# 会话原始需求收集系统设计

## 1. 设计目标
1. 提供一个基于 Web 的原始需求入口与原始需求列表入口。
2. 提供基础账号能力：登录、修改密码。
3. 数据持久化：当前版本使用 SQLite。
4. 最大化复用 Django 内置能力，减少重复开发。

## 2. 技术方案
1. 后端框架：Django
2. 认证能力：`django.contrib.auth` + `django.contrib.sessions`
3. 页面渲染：Django Templates
4. 数据库：SQLite（Django ORM 管理）
5. 管理后台：Django Admin（用于辅助管理会话数据）

## 3. 功能设计
### 3.1 认证
1. 登录页：使用 Django `LoginView`。
2. 退出登录：使用 Django `LogoutView`。
3. 修改密码：使用 Django `PasswordChangeView` + `PasswordChangeDoneView`。
4. 默认管理员：通过数据迁移初始化 `admin / 123456`（仅当用户不存在时创建）。

### 3.2 会话需求收集
1. 会话实体：`RequirementSession`
- `title` 标题
- `created_by` 创建人
- `created_at` 创建时间
- `updated_at` 更新时间
2. 消息实体：`RequirementMessage`
- `session` 所属会话
- `role` 角色（`user` / `assistant`）
- `content` 消息内容
- `created_at` 创建时间
3. 页面能力：
- 会话列表页：展示当前登录用户的聊天窗口（主会话）与总结会话侧栏
- 会话详情页：展示单条会话的多轮消息
- 新建会话：移除聊天场景下的新建会话逻辑，聊天统一写入用户主会话
- 原始需求列表页：展示当前项目 `doc/01-or` 下的文件列表，支持从会话页顶部入口点击跳转
- 路径容错：当当前项目路径无权限或不可访问时，文件列表页回退到系统默认目录 `doc/01-or` 并提示用户，避免页面 500
3. 访问控制：
- 列表/详情/新建必须登录后访问

### 3.3 UI 与交互设计（对话产品风格）
1. 整体布局采用“左侧导航 + 中间对话主区 + 右侧历史会话”三栏结构：
- 左侧：一级/二级菜单导航（聊天、组件、设置等）
- 中间：当前会话内容与输入区
- 右侧：历史会话列表（按最近更新时间倒序）
2. 视觉风格：
- 深色主题、圆角卡片、柔和阴影、低对比边框
- 输入区固定在主内容底部，符合主流大模型对话页输入习惯
 - 消息气泡高度随内容自适应，不设置固定高度
 - 对话内容区设置居中聊天轨道，减少左右气泡视觉距离
3. 交互规则：
- 左侧“聊天”一级菜单保留二级入口“聊天”，并新增二级入口“伙伴”；
- 点击二级“聊天”默认进入当前对话；若当前无激活会话则进入最近一次会话；
- 点击二级“伙伴”进入统一伙伴列表，列表展示伙伴角色属性、能力白名单、知识库关联状态，支持关键词筛选；
- 伙伴列表中每个条目的操作区按“聊一聊 -> 编辑配置”顺序展示；`聊一聊` 需位于“编辑配置”左侧，并以新窗口弹开伙伴专属聊天页；
- 伙伴支持新增与编辑配置，配置项至少包含：角色信息、记忆信息、可调用智能体模块、可调用工具组件模块；知识库路径不再开放编辑；
- 伙伴知识库目录改为系统自动管理：统一落在 `<project.path>/data/companions/<companion_slug>/knowledge`；页面仅展示，不允许手工修改；
- 伙伴列表提供“知识库整理”操作，针对系统自动生成的伙伴知识库目录执行整理并展示报告结果；
- 移除侧栏独立“新对话”按钮，在二级“聊天”项内提供 `+` 新建入口；
- 历史会话列表统一放到右侧栏，点击后切换到对应会话详情；
- 聊天窗口右侧“总结会话”采用右侧抽屉交互，默认收拢为边缘展开入口；点击“展开总结会话”后从右侧抽屉弹出，再次点击、点击遮罩或关闭按钮时收起；
- 一级菜单采用互斥展开（手风琴）交互：点击某个一级菜单展开时，其他一级菜单自动收缩，任一时刻仅保留一个展开分组；默认展开当前常用分组；
- “工具组件”一级菜单默认收起，仅当当前页面属于工具集或工具模块详情时自动展开，避免在聊天页等无关场景干扰导航焦点；
- 一级菜单处于收缩态时，分组卡片高度仅保留标题行，不得在标题下方保留额外空白行或占位内边距；
- 一级菜单收缩态标题行需保持紧凑高度（约单行按钮高度），并清除分组元素默认外边距，避免视觉上出现“多一行空白”；
- 左侧菜单容器在存在剩余高度时，不得对一级菜单分组做纵向拉伸；各分组高度应按内容自适应（收缩态仅单行标题高度）。
- 左侧导航定义为系统级公共菜单：在“设置”分组点击“项目管理”“原始需求列表”等二级菜单后，左侧菜单结构与样式保持不变，不允许切换为页面私有侧栏；
- 左侧导航菜单在条目展开过多导致超出可视区时，菜单区域需启用独立纵向滚动，不得截断底部菜单项；
- 当前页面所属菜单项需高亮显示；当处于设置类页面时，“设置”分组默认展开并保持可见；
- 工具组件相关页面（工具集、组件模块列表、组件 API 测试、组件参数配置）必须显式设置非 `chat` 导航标识，避免与聊天页共用默认值导致“工具组件”二级菜单点击后错误关联到“聊天”一级菜单展开；
- 侧栏底部用户信息区按钮文案保持简短（如“在线”“退出”），避免冗长标签影响视觉密度；
- 新建会话后跳转到对应会话详情页
- 在会话详情页持续发送消息，形成多轮对话
- 未选择会话时展示引导文案和快捷操作
 - 用户发送消息并完成页面刷新后，右侧对话区自动滚动到底部，展示最新用户消息和助手回复
 - 左侧会话列表与右侧对话区独立滚动，互不影响
- 顶部标识“原始需求列表”支持点击跳转到文件列表页
- 文件列表页默认展示当前项目原始需求目录下的文件名、创建日期、更新时间、文件大小（自动换算为 KB/MB 等单位）
- 点击 `.md` 文件名在当前页面加载预览区，展示该文件内容
- 聊天输入框支持 `@` 提及已启用伙伴：输入 `@` 后展示候选列表，支持键盘上下选择并回车插入。
4. 消息展示约束：
- 消息气泡高度必须随内容自适应，不允许固定高度。
- 气泡本身不保留模板缩进空白，避免因空白符导致异常拉高。
- 消息正文渲染需要对模板缩进空白做归一化处理：保留用户真实换行，不保留模板层面的前导/尾随空白。
- 对话轨道宽度需收敛到可读区间（建议 `720~860px`），在大屏下保持居中，避免消息左右跨度过大。
- 单条气泡最大宽度建议不超过轨道的 `78%`，并设置合理最小宽度，保证短消息不显得过窄、长消息不贴边。
- 对话区需设置统一上下留白与行间距，消息间垂直节奏保持一致，减少“松散/拥挤”观感。
 - 语音消息展示规则：若消息包含音频附件，先展示语音播放器；若存在识别文本，默认折叠不展开，用户点击“展开文字/收起文字”按钮后切换可见性。
- 助手消息支持“处理过程轨迹”展示：默认收缩，仅在用户点击“查看处理过程”后展开；
- 处理过程轨迹需按步骤列出，至少覆盖：`思考/规划`、`大模型调用`、`代码或工具组件调用`、`步骤结果`；
- 过程轨迹中的关键步骤默认二级收缩，用户可逐步展开查看详情（输入、输出、错误信息）。
5. 大模型选择器：
- 侧栏使用单一按钮展示当前大模型名称，按钮样式与“当前项目-管理”按钮保持一致。
- 点击按钮展开一级菜单（厂商列表）。
- 鼠标悬停一级菜单项时，在右侧展示二级菜单（该厂商下模型列表）。
- 点击具体模型后即时保存到当前项目，并更新按钮文本为新模型名称。
- 模型按钮与“当前项目”放在同一栏位头部，不再使用独立栏位。
- 左侧栏移除“当前项目”模块，顶部标题栏改为直接展示当前项目名称（位于页面标题右侧），并保留项目管理入口。
- 为避免悬停丢失，二级菜单与一级菜单之间不保留悬停断点，并提高菜单层级，确保可点击。
- 内置阿里模型清单并写入数据库（不存在则补齐），前端直接展示该厂商下全部内置模型供选择。
- 一级菜单字体与现有侧栏风格保持一致（字号、颜色、间距对齐 `chip/meta` 风格），避免视觉突兀。
- 交互实现采用稳定联动：一级厂商列表与右侧二级模型列表同容器双列展示，悬停厂商时右侧即时刷新模型列表。
- 下拉菜单需要以浮层方式显示，层级高于右侧主内容区域，避免二级模型列被页面主体遮挡。
- 布局上保留左侧会话侧栏，模型切换入口放置到主内容顶部左侧区域；仅调整布局，不改变现有视觉主题。
6. 聊天窗口语义约束：
- 对话窗口定位为“当前登录用户聊天记录窗口”，会话列表基于用户维度统一展示；
- 聊天回复不注入伙伴人设，不注入原始需求整理角色约束，使用通用助手语义；
- 发送消息时以上方当前选中的大模型 API 为基准模型；
- 模型上下文默认仅注入当前会话最近 50 条消息，避免长会话导致上下文膨胀；
- 上下文条数支持通过配置项调整，默认值为 50。
7. 总结会话语义约束：
- 右侧“历史会话”改为“总结会话”；
- 总结会话不是独立聊天入口，而是从主聊天记录中抽取整理出的有逻辑片段；
- 提供“从当前聊天消息提炼总结会话”的操作，将目标消息范围内的对话复制沉淀为总结会话；
- 总结会话用于回顾与复盘，不影响主聊天连续上下文。
- 总结会话列表在聊天页默认不展开，不占用主聊天宽度；仅在用户主动展开右侧抽屉时展示。
- 总结会话抽屉宽度在桌面端保持“略宽于默认聊天气泡列”的可读性标准，默认建议约 `360px` 左右，避免标题与元信息过密导致不易识别。
8. 语音输入交互：
- 会话输入区增加麦克风按钮；点击后开始录音，再次点击结束录音。
- 录音文件以音频附件方式随消息一起发送（复用现有附件上传通道）。
- 语音识别采用浏览器内置语音识别能力，将识别文本回填到输入框并随消息提交到后端保存。
- 若浏览器不支持语音识别或识别失败，仍允许用户仅发送音频附件。
9. 输入区布局规范（参考主流对话产品）：
- 输入区采用圆角胶囊风格，单行主输入交互；整体保持低对比边框与浅背景。
- 左侧放置“+”附件入口，点击后触发附件选择。
- 中间为输入区域，占据主要宽度，提示文案简洁（如“有问题，尽管问”）。
- 右侧放置两个操作入口：麦克风按钮（语音识别）与圆形语音按钮（录音开始/停止）。
- 圆形语音按钮在录音中需有明显状态反馈（颜色或文案变化），避免误操作。
- 输入区底部状态行采用单行布局：左侧展示“语音输入状态”，右侧展示“发送”按钮，减少纵向跳动并提升状态可见性。
- 键盘交互默认 `Enter` 直接发送，`Shift + Enter` 换行；在输入法组合态（中文候选上屏）不触发发送。
10. 会话发送与回复更新策略（对齐主流聊天产品）：
- 用户点击发送后，前端立即在对话区渲染本条用户消息，不等待后端完整回复；
- 同时在对话区创建一条助手占位消息（初始可为空或“思考中...”），发送按钮即时切换为“停止”；
- 后端以异步任务方式生成助手回复，前端每 1 秒轮询任务状态接口；
- 轮询返回“部分回复文本”时，前端增量刷新助手占位消息内容；
- 轮询期间若后端新增过程轨迹事件（`process_trace.events`），状态接口需返回最新事件集，前端需在不等待任务结束的情况下即时刷新“中间步骤”展开区；
- 轮询返回“完成/失败/已停止”时，前端结束轮询并将按钮恢复为“发送”；
- 用户点击“停止”后，前端调用停止接口，请求终止当前回复任务；停止后保留当前已生成文本。
- 聊天记录区必须作为独立纵向滚动容器（非整页滚动），在消息超长时可持续下滑查看历史；
- 聊天记录区滚动条需保持可感知（细滚动条样式即可），避免用户误判页面不可继续下滑。
11. 本地模型管理（Ollama / llama-cpp-python）：
- 本地模型操作精简为两个动作：`激活模型` 与 `取消激活`；
- 支持两种本地模型实现模式：`ollama` 与 `llama_cpp`（Python 组件）；
- `ollama` 模式下，`激活模型` 等价于后端串行执行 `pull + warmup`，前端不再暴露“拉取模型/加载模型”两个分离按钮；
- `llama_cpp` 模式下，`激活模型` 等价于后端加载 `model_file_path` 指向的 GGUF 模型到 Django 进程内存，`取消激活` 释放该模型实例；
- 模型操作采用异步后台任务执行，避免请求阻塞；
- 同一条 `LocalLLMConfig` 在存在进行中任务时，后端拒绝重复提交（防止用户反复点击）；
- 前端模型列表页对本地模型运行状态每 1 秒轮询一次状态接口，并基于返回状态动态更新按钮禁用态与状态文案；
- 模型状态需要持久化：任务状态与模型运行状态写入数据库；服务重启后可恢复最近状态展示，并将重启前未完成任务标记为“中断”；
- 状态展示至少包含：`inactive/activating/active/deactivating/failed` 与最近错误信息（若有）。
- 本地模型服务自动启动：当配置的 Ollama endpoint 为本机地址（如 `127.0.0.1/localhost`）且服务不可用时，后端自动尝试执行 `ollama serve` 拉起服务；
- 启动策略采用“双通道”：Django 启动阶段按已启用配置预热拉起 + 运行时按需兜底拉起；用户不需要手工启动 Ollama 服务。

### 5.8 表：collector_localllmconfig（补充字段）
1. `runtime_backend`：本地模型实现模式（`ollama/llama_cpp`，默认 `ollama`）
2. `endpoint`：Ollama 地址（`runtime_backend=ollama` 时必填）
3. `model_name`：模型标识（两种模式通用）
4. `keep_alive`：Ollama keep_alive（`runtime_backend=ollama` 生效）
5. `model_file_path`：GGUF 模型文件路径（`runtime_backend=llama_cpp` 时必填）
6. `llama_cpp_n_ctx`：llama-cpp 上下文窗口（`runtime_backend=llama_cpp` 生效）

### 3.4 AI 回复策略（当前版本）
1. 会话采用单段式聊天处理：
- 每次发送消息时，后端从当前会话截取“最近 N 条消息（默认 50）”作为上下文；
- 不再执行“收集 -> 整理”阶段切换，不再在聊天链路中自动生成原始需求文档；
- 不注入人设或伙伴角色提示词，按通用助手模式生成回复。
2. 模型选择策略：
- 主聊天会话：聊天请求以上方当前选择的大模型 API 作为基准模型；若未选定项目模型，则回退到系统默认模型配置。
- 伙伴聊天会话：不提供聊天窗口内模型切换；发送消息时自动使用该伙伴的默认模型（`CompanionProfile.default_model_name`），未配置时再回退到项目模型与系统默认模型。
- 伙伴聊天窗口顶部不展示大模型切换器，避免与伙伴默认模型策略冲突。
3. 降级策略：
- 大模型不可用时，返回可读错误提示，不覆盖既有会话记录。
4. 模型能力兼容策略：
- 默认按文本对话调用；当存在可识别多模态附件时，按“所选模型 + 附件模态”进行路由。
- 当项目选择的模型为视觉模型（如 `qwen-vl-*`）且本次请求未构造多模态输入时，后端自动回退到文本模型（默认 `qwen-plus` 或系统配置模型）执行。
- 回退逻辑仅影响模型调用，不改变项目已选模型配置；并在日志中记录回退信息，便于排查。
5. 多模态消息构造规则（图片/音视频附件）：
- 会话消息在数据库中继续保存 `content` 与 `attachment`；构造模型请求时按“文本 + 附件”组装。
- 当消息包含可识别多模态附件（如 `image/*`、`video/*`、`audio/*`）且本次调用目标模型为视觉模型（`qwen-vl-*`）时，后端将该消息构造成多模态 `content` 列表：
  - 文本块：`{"text": "..."}`
  - 附件块需按模态字段构造（不使用通用 `file`）：
    - 图片：`{"image": "data:<mime_type>;base64,<encoded_content>"}`
    - 视频：`{"video": "data:<mime_type>;base64,<encoded_content>"}`
    - 音频：`{"audio": "data:<mime_type>;base64,<encoded_content>"}`
- 当消息仅有附件无文本时，仍允许发送附件块；当仅有文本无附件时，按纯文本消息发送。
- 若附件类型不属于视觉模型支持的模态（如 `pdf/doc/docx`），该附件不构造成视觉模型附件块，仅作为文本提示参与上下文。
6. 多模态与模型路由联动策略：
- 路由判断基于“所选模型 + 本次请求是否存在**可识别多模态附件块**”：
  - 视觉模型 + 存在可识别多模态附件块：保持视觉模型，不回退。
  - 视觉模型 + 不存在可识别多模态附件块：回退到文本模型（`qwen-plus` 或系统配置模型）。
  - 文本模型：始终按文本调用（附件不触发视觉路由）。
- 聊天模式统一使用上述联动策略，保证日志与实际调用一致。
8. 伙伴能力注入策略（伙伴聊天会话）：
- 伙伴聊天在构造模型上下文时注入伙伴能力约束：角色信息、记忆信息、可调用智能体模块（心法/号令）与可调用工具组件模块；
- 对伙伴配置的智能体模块，按模块枚举可用项（README/目录）并注入“可用能力清单”，保证“已配置即可用”；
- 伙伴聊天需同时注入项目规则与系统规则/技能摘要，要求回复遵循对应规则；
- 当需要调用工具或组件能力时，仅允许在伙伴配置白名单范围内声明和使用。
9. 伙伴授权分层策略（新增）：
- 伙伴授权从“仅模块级”升级为“四层收敛”模型：`工具集` -> `组件模块` -> `组件` -> `组件 API`；
- 工具集授权用于限制伙伴可见与可声明的工具集能力范围，至少支持 `tools.manager.list_toolsets()` 返回的工具集项；
- 组件模块授权继续保留（如 `observe/handle/decide/communicate`），作为第一道运行时拦截；
- 组件授权用于限制具体 `component_key`（如 `handle.macos_terminal_component`），用于模块内细分；
- API 授权用于限制具体函数路径（如 `component.observe.understand_current_screen`），作为最细粒度拦截；
- 授权校验采用“由粗到细逐层收敛”：
  - 若某层未配置（空），表示该层不额外收窄；
  - 若某层已配置（非空），必须命中该层白名单；
  - 任一层不命中即拒绝执行并返回可读错误（包含拒绝层级与目标标识）。
10. 工具集授权运行时边界（修订）：
- 工具集授权用于约束 `tools.*` 工具集执行链路；
- `component.*` 直调链路（`mindforgeRunner/toolRunner -> ComponentCallTool`）不因“未命中工具集项”直接阻断，避免误把 `component_call_tool` 作为业务授权门槛导致“已授权仍无调用”；
- 组件调用的核心安全约束继续由“组件模块/组件/API 白名单 + 组件系统权限校验”负责；
- 当后续引入 `tools.*` 计划步骤时，再按目标 toolset key 执行强校验。
9. 伙伴聊天执行架构（自主规划 + 多智能体协同）：
- 伙伴聊天不再采用“单次大模型对话直接回复”模式，改为“规划 -> 执行 -> 汇总”三段式执行；
- 规划阶段由 `CustomPlanner` 基于 LLM 动态输出结构化执行计划（步骤类型、输入、目标、依赖关系），不再固定写死步骤模板；
- 规划阶段支持步骤上限约束 `max_plan_steps`（默认 8），用于限制单次任务执行复杂度与耗时；
- 规划输出必须通过协议校验（步骤类型白名单、目标白名单、依赖合法性）；校验失败时回退到规则模板计划，保证可执行性；
- 执行阶段由 `Coordinator` 调度多个执行器：`mindforgeRunner`、`helmRunner`、`toolRunner`；
- `mindforgeRunner` 负责调用心法（如 `agents/mindforge/react_engine`）执行推理与工具链路；
- `helmRunner` 负责号令类能力（如会话阶段决策与流程命令）；
- `mindforgeRunner` 负责 ReAct 组件调用主链路（思考 -> 选工具 -> 调用组件 -> 观察 -> 再规划）；
- `toolRunner` 仅用于少量“可直接调用”的安全组件函数（如系统信息/屏幕理解/文本生成）；其余组件调用统一回到 `mindforgeRunner`；
- 汇总阶段由 `AnswerSynthesizer` 基于多步骤执行结果生成最终回复，持续写回助手占位消息；
- 工具步骤采用“失败感知重规划”策略：首选工具失败后不得无脑遍历候选工具；应先基于失败类型判断是否可恢复，再决定是否尝试替代路径；
- 对“未授权/越权/白名单拒绝”类错误视为不可恢复错误，当前步骤立即停止重试，禁止继续尝试其他工具；
- 对“超时/连接失败/临时执行异常”类可恢复错误，允许最多 1 次强相关替代尝试（同模块或同组件能力域，且必须通过授权校验）；
- 当 `toolRunner` 直调失败或命中“非直调白名单组件函数”时，协调器触发一次 `mindforgeRunner` 重规划执行，避免继续在 `toolRunner` 内盲试；
- 工具替代重试信息需写入过程轨迹（尝试顺序、失败原因、最终命中工具），用于前端展示与排障；
- 执行依赖采用“硬依赖 + 软依赖”混合策略：工具执行等关键步骤遵循硬依赖；`summarize` 汇总步骤采用软依赖，即使上游存在失败步骤也必须执行；
- 任一执行步骤失败时不直接触发全局降级；应继续执行可执行步骤，并在汇总阶段明确暴露失败项与替代建议；
- 仅当编排入口异常、或汇总阶段无法生成可用回复时，才回退到通用 `analyzer.chat`，确保会话可回复不中断。
12. mindforge 候选工具发现机制（动态）：
- `mindforgeRunner` 构建 ReAct 工具清单时，不允许在代码内硬编码候选工具列表；
- 候选工具必须通过 `ComponentCallTool.control_call("component.list_components", ...)` 查询组件管理层返回结果动态生成；
- 仅保留满足以下条件的函数作为候选工具：组件当前 `enabled=true`、函数路径以 `component.` 开头、模块名在伙伴 `allowed_control_modules` 白名单内；
- 工具展示名可由函数路径派生（如 `component.observe.understand_current_screen -> observe_understand_current_screen`），描述优先使用组件管理层元数据，缺失时再使用通用降级文案；
- 当管理层查询失败或白名单下无可用函数时，返回明确错误并走现有降级回复链路，禁止回退到写死候选工具常量。
13. 智能体/工具/组件能力检索与缓存机制（匹配准确率优化）：
- 新增统一能力检索层，聚合 `agents`、`tools`、`component` 三类元数据，按“描述信息 + 标识信息”构建可检索索引；
- 索引数据持久化写入 `data/database/capability_callable_index.json`，并保留进程内热缓存；
- 缓存内容至少包含：`kind`（agent/tool/component_function）、`name`、`path`、`module`、`description`、`search_text`；
- 组件描述信息统一通过 `ComponentCallTool.control_info()` 读取 `component/**/README.json` 聚合结果，禁止在编排层绕过工具直接导入 `component`；
- 智能体与工具描述信息分别通过 `agents.manager` 与 `tools.manager` 获取，避免页面层/编排层重复扫描目录；
- Django 服务启动时执行能力索引预加载：优先加载磁盘持久化文件到内存；若文件缺失或失效则重建并落盘；
- 能力索引需维护源数据签名（至少覆盖 `agents`、`tools`、`component` 目录关键文件）；当检测到工具/智能体/组件发生更新时自动重建索引；
- 重建流程采用“内存与磁盘双写同步”策略，保证更新后内存索引与持久化文件一致，禁止仅更新单侧缓存；
- 检索模式支持三种：
  - 传统检索（`traditional`）：关键词匹配与词项重叠得分；
  - 向量检索（`vector`）：基于本地词向量（TF-IDF + 余弦相似度）计算语义相似度；
  - 混合检索（`hybrid`）：融合关键词得分与向量得分作为默认策略；
- 伙伴编排入口允许携带 `capability_search_mode`（默认 `hybrid`），用于规划器与 mindforge 工具候选排序；
- 规划阶段在选择工具函数时优先使用检索结果（限定伙伴白名单模块），仅在无命中时回退到既有关键词兜底规则；
- `mindforgeRunner` 在构建候选工具清单时按检索得分排序，并注入函数描述，提升 LLM 工具选择准确率；
- 检索层仅负责“发现与排序”，不放宽白名单、组件启停、权限校验等既有安全约束。
14. Django 平台搜索接入 zvec（易用性 + LLM 测试）：
- Django 平台中的三类能力查询页（智能体项、工具集、组件 API）统一接入能力检索层，并支持 `zvec` 向量检索引擎；
- 页面请求支持检索引擎参数（`search_engine`）：`auto/native/zvec`，默认 `auto`；
- 页面请求支持检索策略参数（`search_mode`）：`traditional/vector/hybrid`，默认 `hybrid`；
- `search_engine=zvec` 时优先走 zvec 检索；当 zvec 不可用或索引异常时自动降级到 native 检索，页面显示降级提示但不报错；
- zvec 索引文件持久化到 `data/database`，并与能力索引源签名绑定；智能体/工具/组件变更后自动重建，保持与内存索引一致；
- 三类查询页在搜索结果中展示“检索命中信息”（得分、命中路径、摘要），便于人工验证与 LLM 检索测试；
- 搜索页提供“LLM 检索测试视图”：显示本次 TopN 命中结果与核心评分字段，便于校验提示词与检索策略效果；
- 页面搜索交互保持向后兼容：不传 `search_engine/search_mode` 时行为与历史一致。
10. 执行计划协议（MVP）：
- 计划输出采用结构化 JSON，最小字段包括：`plan_id`、`goal`、`steps`、`final_strategy`；
- 每个步骤最小字段包括：`step_id`、`step_type`（`reason/agent/tool/summarize`）、`target`、`input`、`depends_on`；
- 执行结果最小字段包括：`step_id`、`status`（`success/failed/skipped`）、`output`、`error`、`duration_ms`；
- 计划协议新增约束字段：`max_plan_steps`（编排步数上限，默认 8）；
- 规划器在落地执行前必须裁剪步骤总数到 `max_plan_steps`，并补齐末尾 `summarize` 步骤；
- 工具步骤 `input` 可选扩展字段：`fallback_targets`（候选替代路径列表）；规划器仅保留与主目标强相关且已授权的候选项，不允许自动拼接跨域无关工具；
- `tool` 计划步骤只允许直调白名单函数；高耦合/高上下文组件函数（如终端会话类）必须交由 `mindforge` 链路决策调用；
- `summarize` 步骤的执行语义：默认不因 `depends_on` 中任一步骤失败而被跳过，执行器需基于“已成功 + 已失败”步骤集合生成可读最终答复；
- 聊天任务状态接口保持兼容，新增扩展字段（可选）`plan_steps`、`active_agent`、`tool_events`。
11. 过程轨迹记录与展示：
- 异步回复任务需持久化 `process_trace`，用于页面刷新后回放执行过程；
- `process_trace` 建议包含：`events[]`，每个事件最小字段为 `kind/title/status/input/output/error/ts`；
- 轮询状态接口返回 `process_trace`，前端按“总览折叠 + 步骤折叠”方式渲染；
- 过程轨迹写入策略采用“关键步骤增量持久化”：每次追加阶段事件后立即写回 `collector_chatreplytask.execution_trace`，保证轮询可见；
- 过程轨迹仅用于可观测与排障，不参与大模型上下文拼装。
7. 模型响应解析兼容规则：
- 若响应同时包含 `text` 与 `choices[*].message.content`，且 `text` 为空（`null` 或空串），应优先使用 `choices` 中的消息内容。
- 当 `content` 为数组时，按顺序提取其中的 `text` 字段并拼接；当 `content` 为字符串时直接使用。
- 解析层必须保证返回字符串类型，避免 `None` 进入后续 `strip()` 引发异常。

### 3.5 规则读取与优先级设计
1. 新增通用工具 `tools/rule_reader`，供会话整理等模块复用。
2. 输入可以是目录或文件路径：
- 若输入为文件，从其所在目录开始向上逐级查找 `rules.md`。
- 若输入为目录，从该目录开始向上逐级查找 `rules.md`。
3. 层级扫描范围：
- 默认向上扫描到文件系统根目录；
- 支持传入 `stop_at`（建议传项目根目录）限制扫描上边界，避免跨项目读取。
4. 优先级规则：
- 距离目标路径最近的 `rules.md` 优先级最高；
- 返回结果按“高优先级 -> 低优先级”排序（即“近 -> 远”）。
5. 输出结构：
- `rules`：规则列表（每项包含路径、层级距离、内容）；
- `merged_rules`：可直接注入提示词的拼接文本（按优先级顺序分段）；
- `resolution_order`：优先级说明文本，明确“就近优先”。
6. 在需求整理阶段，优先使用 `merged_rules` 注入 `system` 提示词，确保模型按项目规则输出原始需求文档。

### 3.6 本地屏幕理解组件（component/observe）
1. 目标：
- 在 `component/observe` 下提供一个轻量、本地可运行的“截图 + 视觉理解”组件；
- 不依赖云端服务，依赖本机截图库与本地 vLLM 服务闭环完成推理；
- 适配消费级显卡（RTX 3060/4060）场景，优先小参数视觉模型（如 `Qwen2-VL-0.5B-Instruct`）。
2. 模块划分：
- `component/observe/screen_capture_component/api.py`：屏幕截图能力，默认使用 `mss`，支持区域截图与全屏截图；
- `component/observe/screen_understand_component/qwen2vl_vllm.py`：vLLM 推理客户端，调用本地 OpenAI 兼容接口（`/v1/chat/completions`）；
- `component/observe/screen_understand_component/api.py`：端到端编排（截图 -> 图片编码 -> 视觉理解）；
- `component/observe/__init__.py`：对外函数式 API 唯一导出入口；
- `component/observe/screen_capture_component/README.md`：截图组件说明；
- `component/observe/screen_understand_component/README.md`：理解组件说明。
3. 推理链路：
- 输入可为“截图路径”或“图像对象（PIL.Image）”；
- 若未提供图像输入，可直接执行实时截图；
- 将图像编码为 `data:image/png;base64,...` 后发送给本地 vLLM；
- 返回自然语言描述，或按 `json_mode=True` 返回结构化 JSON。
4. 本地 vLLM 部署约束：
- 使用 vLLM 启动本地模型服务，推荐命令：
  - `python -m vllm.entrypoints.openai.api_server --model Qwen/Qwen2-VL-0.5B-Instruct --served-model-name qwen2-vl-0.5b --max-model-len 4096 --dtype auto --gpu-memory-utilization 0.85`
- 默认访问地址：`http://127.0.0.1:8000/v1/chat/completions`；
- 组件通过可配置参数支持切换 `base_url`、`model`、`timeout`。
5. 性能与资源策略（轻量化）：
- 截图优先使用 `mss`（低开销、速度快），默认输出 PNG；
- 推理参数默认偏保守：`temperature=0.2`、`max_tokens=256`；
- 支持限制截图区域，减少图像尺寸与显存占用；
- API 采用懒初始化，不在 import 时加载重依赖。
6. 异常与边界处理：
- 本地 vLLM 不可用时抛出可读错误（连接失败、超时、非 200 响应）；
- 截图库缺失时提示安装依赖；
- 图片输入为空、路径不存在、JSON 解析失败时返回明确异常信息；
- 对模型返回进行兜底：优先取文本，`json_mode=True` 时先尝试 JSON 解析，失败则返回原始文本。
7. Demo 运行约定：
- `component/observe` 外层目录不再提供聚合 demo；
- 各组件在自身目录独立提供 `demo.py`（如 `screen_capture_component/demo.py`、`screen_understand_component/demo.py`）；
- 默认通过文件内变量配置运行参数，不依赖命令行参数；
- 组件 demo 必须使用绝对导入，确保可直接执行；
- Demo 仅用于开发验证，不参与 Django 主流程。
8. tools 工具封装约定：
- `tools/control_call_tool` 作为面向 `component` 的通用工具封装；
- 工具在调用截图/理解能力前，必须先读取并检查自身 `requirements.txt` 依赖；
- 若发现依赖缺失，工具先执行安装，再继续调用 `component` 层功能；
- 支持 `component` 整体信息查询（读取 `component/**/README.json`）；
- 支持特定函数路径调用，统一入口命名为 `control_call`；
- 工具返回统一字典结构：`success`、`data`、`error`。
9. 字符串路径调用约定：
- 新增 `component` 统一调用入口，支持通过字符串函数路径调用，例如：`component.observe.understand_current_screen`；
- 调用器负责完成“路径解析 -> 函数加载 -> 参数透传 -> 异常抛出”；
- 为降低误调用风险，函数路径必须以 `component.` 前缀开头。
10. component 目录自述约定（JSON）：
- `component` 及其每个子目录都必须包含一个 JSON 自述文件（`README.json`）；
- 自述文件需明确列出“函数路径 + 功能描述 + 入参/出参说明”；
- 当新增/调整函数时，同步更新对应目录 `README.json`。
- 依赖包导入名映射（如 `pillow -> PIL`）统一配置在 `component/README.json` 的工具规则字段中；
- tool 侧仅负责读取该规则并执行逻辑，不再新增独立映射配置。
12. component 调用规范去模糊：
- 不保留模块级兼容转发文件（如 `screen_capture.py` / `screen_understand.py`）；
- 对外调用仅使用 `README.json` 与 `component_index.json` 登记的规范函数路径；
- 升级时通过文档与索引显式发布变更，不通过隐式别名兜底。
11. Django 首页组件菜单与函数管理：
- 在首页左侧栏增加树形菜单：一级菜单 `组件（component）`，二级菜单按顺序展示 `沟通（communicate）`、`观察（observe）`、`决策（decide）`、`操作（handle）`；
- `component` 功能页左侧菜单不再单独实现，必须复用与会话页一致的侧栏菜单模板（单一引用），保证菜单结构、文案、交互（展开/收缩）一致；
- 点击二级菜单进入对应功能页，列表展示该模块函数信息（读取 `component/<module>/README.json`）；
- 功能页支持函数信息模糊搜索（函数路径、描述、输入/输出文本匹配）；
- 功能页支持模块目录下载：将 `component/<module>` 目录打包为 zip 下载；
- 功能页支持模块目录上传：上传 zip 后在后端解压覆盖到对应目录，并基于最新 `README.json` 刷新函数信息；
- 上传解压需进行路径安全校验，禁止 zip slip（目录穿越）写入目标目录外路径。
12. 伙伴管理（Companion）：
- 伙伴作为独立业务实体（更接近用户角色），不与 `agents` 目录结构绑定；
- 伙伴通过配置关联能力白名单（智能体模块、工具组件模块），由运行时按白名单调用；
- 伙伴维护独立记忆字段与知识库路径，知识库整理能力复用现有 `KnowledgeCurationTool`。
- 伙伴新增/编辑页面支持纵向滚动，避免内容超出可视区后无法继续编辑；
- 伙伴模块白名单选择区域提供“全选/清空”快捷操作，降低配置成本。
- 伙伴新增/编辑页面新增模型配置区：支持在“API 模型”与“本地模型”二选一。
- 当选择“API 模型”时，必须选择一条可用的 `LLMApiConfig`；当选择“本地模型”时，必须选择一条可用的 `LocalLLMConfig`。
- 伙伴模型手动配置采用单行三级联动：一级选择模型来源（API/本地），二级选择可用模型配置（对应来源下可用配置），三级选择具体模型与参数版本（下拉）。
- 默认模型候选项需与“模型来源 + 二级配置”联动刷新：选择 `LLMApiConfig` 时展示该 API 配置下可用模型；选择 `LocalLLMConfig` 时展示该本地配置的模型与参数版本。
- 当切换模型来源或二级配置时，三级候选项需即时刷新；若原值不在新候选集中，需清空并提示重新选择。
- 伙伴支持配置“备用模型”（多选），候选项可同时来自 API 与本地模型；备用模型不得与已选默认模型重复。
- 伙伴支持“自动模式”：启用后不需要配置手动三级联动与备用模型，运行时由系统自动决策模型。
- 伙伴列表页展示模型来源、绑定配置名称与默认模型，便于快速核对伙伴运行配置。
- 模型配置校验规则：保存时必须保证“来源 + 对应配置”一致，不允许出现来源与配置类型不匹配的脏数据。
- 伙伴详情配置页权限区采用“分组卡片 + 紧凑勾选”样式，避免多层滚动与大面积空白导致难以阅读。
- 伙伴详情页移除侧栏“知识库操作”区，仅保留顶部“执行知识库整理”按钮；整理结果通过页面消息提示反馈，不在详情页右侧长期占位展示。
- 伙伴详情配置页在长表单场景下必须保留主内容区纵向滚动条（`.main-panel` 可滚动），不得因样式调整导致页面无法下滑。

## 5. 数据库设计

1. `id`：主键，自增
2. `title`：字符型，最大 200

4. `created_by_id`：外键，关联 `auth_user.id`

6. `updated_at`：更新时间，自动更新

1. `id`：主键，自增
2. `session_id`：外键，关联 `collector_requirementsession.id`

4. `content`：文本型
   - 语音消息场景下，保存语音识别后的文本（允许为空，表示仅发送语音附件）
5. `created_at`：创建时间，自动写入

- 用户名：admin

## 6. 安全与约束
1. 登录后访问控制使用 Django 装饰器/Mixin。
2. 密码修改后继续保持登录状态（默认行为）。
3. 开发环境允许 `DEBUG=True`，生产需关闭并配置安全项。
4. 伙伴编排安全约束：
- 伙伴未配置的心法/号令模块不可调度；
- 伙伴未配置的工具集不可查询、不可声明、不可执行；
- 伙伴未配置的工具组件模块不可调用；
- 伙伴配置了组件白名单时，未授权组件不可调用；
- 伙伴配置了组件 API 白名单时，未授权 API 不可调用；
- 工具调用函数路径必须以 `component.` 前缀开头；
- 工具入参必须通过 JSON 结构校验，拒绝危险路径和越权参数。
